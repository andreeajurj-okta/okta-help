Atlassian

このドキュメントでは、Okta Integration Networkの既存のConfluenceおよびJiraCloudプロビジョニングアプリ（Atlassian Jira Cloud、Jira（Atlassian）、およびAtlassian Confluence Cloud）から新しいAtlassian Cloud OINアプリに移行する方法について説明します。

概要

SAMLSSOおよびSCIMプロビジョニング機能をサポートするOktaIntegration Network（OIN）（Atlassianによって構築および保守）での新しい Atlassian Cloud アプリ統合の最近のリリースに伴い、OktaはAtlassianCloudのお客様がこの新しいアプリケーションに移行することを推奨しています。次のアプリ統合から。

  • Atlassian Confluence Cloud：ConfluenceCloudアプリで使用される非推奨のSOAPAPIがAtlassianによって削除されたことに伴い、プロビジョニング機能が非推奨になりました。プロビジョニングを継続して使用するには、AtlassianCloudへの切り替えが必須です。
  • Atlassian Jira Cloud：このアプリ統合のプロビジョニング機能は廃止されていません。必要に応じてこの統合を引き続き使用できますが、お客様は新しいAtlassianCloudアプリに移行することをお勧めします。
  • Jira（Atlassian）：このアプリ統合のプロビジョニング機能は廃止されていません。必要に応じてこの統合を引き続き使用できますが、お客様は新しいAtlassianCloudアプリに移行することをお勧めします。

覚えておいてください：組織でアプリを構成するときに、これらのアプリのデフォルト名を変更した可能性があります。

新しい Atlassian Cloud アプリの統合は、SAMLを有効にし、上記のすべてのAtlassian Cloud製品（およびそれ以上）のSCIMプロビジョニングを介してユーザーを管理するためのワンストップショップです。組織レベルでSAMLをセットアップし、グループとそのメンバーをAtlassian Cloudにプッシュできるようにすることで、拡張機能と優れたユーザーエクスペリエンスを提供します。新しいアプリの統合は将来の拡張機能を受け取り、Atlassianによって維持されるため、お客様はできるだけ早い機会にアップグレードすることをお勧めします。

注：これらのアプリ統合でConfluence / Jiraサーバーを使用している場合、アクションは必要ありません。移行は、クラウド製品を使用しているお客様にのみ適用されます。

移行の計画
  • 計画、テスト、展開を含め、プロジェクトに少なくとも1〜2週間の予算を立てます。
  • 以下の移行ドキュメントを確認して、使用可能なSSOおよびSCIM機能を理解してください。
  • OINで利用可能な新しいAtlassianCloud統合をセットアップしてテストします。
  • EOLの日付より前に、すべてのユーザーを新しいAtlassian Cloud統合に移行します。

Atlassian Confluence CloudとAtlassianJira Cloudの統合のEOLに関する質問については、 support.okta.comにお問い合わせください。Atlassianの製品、新しいAtlassian Cloudの統合または移行に関する質問については、Atlassianサポートにお問い合わせください。

要件

  • Atlassian Cloudアプリを利用し、SAML機能とSCIM機能の両方を使用するには、Atlassian Accessサブスクリプションが必要です。詳細については、 https://www.atlassian.com/software/accessを参照してください。
  • Oktaで構築されたConfluence / Jiraアプリケーションを使用すると、サイトレベルでユーザーを管理できます。Atlassianで構築されたAtlassian Cloudアプリケーションを使用すると、組織レベルでユーザーを管理できます。Atlassianのドキュメントによると、Atlassian組織は複数のサイトを持つことができ、顧客が製品とユーザーを管理するための一元化された場所です。以下の移行手順に進む前に、Jira / ConfluenceサイトがAtlassian組織に追加されていることを確認してください（詳細については、 https://confluence.atlassian.com/cloud/set-up-an-を参照してください）。 atlassian-organization-938859734.html）。
  • 本番データに進む前に、サンプルユーザー/グループを使用して移行フローをテストし、すべてが正しく機能していることを確認する必要があります。
  • Atlassian Cloudアプリケーションにユーザーを割り当てるときに、個別の割り当てを使用することはお勧めしません。以下の移行手順全体で説明するように、ユーザーの割り当ては、製品アクセスがグループに付与されるため、グループ割り当てを介して行う必要があります。これらのグループのメンバーは、所属するグループに付与された製品アクセス権を自動的に取得します。OktaのAtlassian Cloudアプリケーションを介してユーザーをプッシュするときに個別のユーザー割り当てを使用する場合、そのユーザーをグループに追加するまで、そのユーザーは製品にアクセスできません。
  • Oktaで構築されたJiraアプリケーションを使用して以前にプッシュしたAtlassian Cloudアプリケーションを介してグループをプッシュしている場合、グループは自動的にリンクされ、そのグループに付与した製品アクセスはすべて同じままである必要があります。
  • Jira / Confluenceアプリケーションを介してプッシュしたユーザーを同じグループの一部にする場合は、Atlassian Cloudアプリケーションを介してプッシュする前に、ユーザーをOktaのそれらのグループに割り当てる必要があります。

    注：Oktaで構築されたJira / Confluenceアプリケーションとは異なり、AtlassianCloudアプリは新しいユーザーをプッシュするときにグループ検出をサポートしません。

  • Atlassian Cloudアプリの場合、ユーザーアカウントは、Atlassian組織を使用して検証されたドメインを使用している場合にのみプッシュできます。Atlassian組織レベルでは、ドメインを使用してアカウントをプッシュおよび管理できるように、ドメインを所有していることを確認する必要があります。

  • Atlassianサイトレベルでは、アクセスを許可する任意のドメインを持つユーザーをサイトに追加できます。Atlassian組織レベルのように、ドメインを所有していることを確認する必要はありません。

  • 移行するときは、次の点に注意してください。組織レベルで確認できないドメインを使用しているサイトレベルのユーザーがいる場合、AtlassianCloudアプリを使用してそのユーザーをプッシュすることはできません。

移行手順

Atlassian Cloud と呼ばれる新しいAtlassianアプリケーションがOkta統合ネットワーク（OIN）に追加され、Oktaの顧客により良い全体的なエクスペリエンスを提供します。この新しいアプリケーションは、Atlassianの新しいIDプラットフォームのライフサイクル管理サポートを追加します。

これらの更新を利用するには、Okta組織にAtlassian Cloudの新しいインスタンスを追加する必要があります。Oktaで構築されたJira / Confluence Cloudアプリケーションのいずれかを以前に追加した場合は、以下の手順に従って、これらのアプリケーションから新しいAtlassianCloudアプリケーションに移行します。

  1. Okta組織に管理者としてサインインします。

  2. 管理コンソールを開く。

  3. ［Add Application（アプリケーションの追加）］をクリックします。

  4. Atlassian Cloudの新しいインスタンスを追加します。

  5. 使用する機能（SWA、OMM、SAML、プロビジョニング）に応じてアプリケーションを構成します。

    1. SWA： サインオン タブで、サインオン方法としてSWAを選択し、ユーザー資格情報を保存するための目的のオプションを選択して、 保存：をクリックします。

      1. 古いJira / Confluence CloudアプリケーションにSWAサインオンモードを使用していた場合は、新しいAtlassianCloudアプリケーションに割り当てられるすべてのユーザーのクレデンシャルを再入力する必要があります。注：ユーザーがパスワードを取得する必要がある場合は、以下の手順に従って取得できます。

        • Oktaのホームページで、古いJira / Confluence Cloudアプリケーションにカーソルを合わせ、歯車のアイコンをクリックします。

        • ] [パスワードを参照 タブで、[ パスワードを表示]]をクリックします。ユーザーは、資格情報を確認するために再認証するように求められます。

        • 古いJira / Confluence Cloudアプリインスタンスを非アクティブ化/削除する前に、Jira / Confluence Cloudを介してパスワードを再設定しないように、パスワードを取得する必要があるすべてのユーザーがそうしていることを確認してください。

      2. 既存のJira/Confluence appアプリと同じ［Application Username Format（アプリケーション ユーザー名の形式）］が選択／追加されていることを確認します。

      3. ［Sign On Policy（サインオン ポリシー）］セクションを下方にスクロールします。古いアプリ用に構成した方法で、すべてのサインオンポリシーを新しいアプリにコピーします（詳細については、 アプリケーションのサインオンポリシーの追加 を参照してください）。

    2. OMM： Mobile タブで、ユーザーが利用できるようにしたいすべてのモバイルアプリケーションをOkta Mobile AppStoreでダウンロードできるようにします。

      注：古いJira / ConfluenceCloudアプリケーションでJira / Confluence Cloud OMMアプリケーションをアクティブ化した場合は、新しいAtlassian Cloudアプリケーションを追加した後、これらを再度アクティブ化する必要があります。

    3. SAML： サインオン タブで、サインオン方法として SAML を選択します。 [セットアップ手順の表示] [ をクリックし、すべての手順に従ってAtlassian CloudアプリのSAMLを構成します。

    4. SCIM： Atlassian Cloud SCIM構成ガイドに概説されている手順に従います。Atlassian Cloud SCIMアプリを介したユーザープロビジョニングは、グループ割り当てを介して実行する必要があることに注意してください。ユーザー製品へのアクセスは、グループを介して割り当てられます。

      Atlassian構成ガイドに記載されていない一般的なプロビジョニングシナリオ

      • Oktaで構築されたJiraアプリを使用してプッシュされた既存のグループをプッシュする

        セクションで説明したように、以前にOktaで構築されたJiraアプリを使用してプッシュされたグループをAtlassian Cloudアプリにプッシュすると、グループが正しくリンクされるはずです。これらのグループに付与された製品アクセスはすべて同じままです。同じグループをプッシュするために特別な手順は必要ありません。

      • ルールを使用してグループをプッシュする

        Oktaで構築されたJira / Confluenceアプリでグループを自動的にプッシュするルールを以前に設定した場合は、Atlassian Cloudアプリインスタンスに同じルールを追加してください。

        これは、外部ソース（Active Directory、LDAPなど）からグループをインポートして、それらのソースから作成されたグループが引き続き自動的にプッシュされるようにする場合に重要です。

      • ユーザーをプッシュするときにカスタムマッピングを使用する

        Atlassian Cloudアプリは現在、Second EmailおよびMobile Phone属性をサポートしていません。これらの2つの属性は、Oktaで構築されたJira / Confluenceアプリでサポートされていました。他のすべての属性がサポートされています。Atlassian Cloud SCIMアプリのデフォルトの属性とマッピングの完全なリストについては、 Atlassian Cloud SCIM構成ガイドを参照してください。SCIMアプリで属性を追加/削除する必要がある場合、古いJira / Confluence属性マッピングと一致するように任意の属性へのマッピングを更新する場合は、次のガイドに従ってください。 アプリケーションの属性とそれに対応するマッピングを確認する。

        注：ユーザーを組織レベルでプッシュすると、Atlassian Cloudアプリを使用して設定された属性マッピングは、Oktaで構築されたJira / Confluenceアプリを使用して設定された属性マッピングを上書きします。

  6. 必要なすべての機能を有効にしたら、新しいAtlassian Cloudアプリケーションの 割り当て タブに移動します。 [割り当て]をクリックして、古いJira / ConfluenceCloudアプリケーションに割り当てられているのと同じユーザー/グループの割り当てを開始します。

    重要

    1. すべてのユーザーを新しいAtlassian Cloudインスタンスに割り当てて、ユーザーの偶発的なプロビジョニング解除やアクセスの喪失を回避してください。

    2. プロビジョニングを有効にする場合は、ユーザーをアプリケーションに割り当てる前に、Atlassian Cloud SCIM構成ガイドを確認することが重要です。正しい製品アクセス許可でユーザーを適切にプロビジョニングするには、ユーザーをグループ割り当てで割り当てる必要があります（これらのグループは割り当て前に最初にプッシュする必要があります）。

  7. 管理コンソールに戻ります。

  8. Jira / ConfluenceCloudアプリケーションを開きます。

    注：これは、手順4で新しいアプリケーションを追加する前に追加した前のアプリケーションです。

  9. オプション：以前にJira / Confluence Cloudアプリにプロビジョニングを使用した場合：

    1. ［Provisioning（プロビジョニング）］タブをクリックします。

    2. 設定 で、 API統合を選択します。

    3. [編集]をクリックし、[API統合を有効にする]のチェックボックスをオフにします。

    4. ［Save（保存）］をクリックします。

  10. これで、古いJira / Confluence Cloudアプリケーションを非アクティブ化または削除して、追加した新しいAtlassianCloudアプリケーションを引き続き使用できます。ただし、アプリを短期間（1〜2週間）非表示にし、古いJira / Confluenceアプリを非アクティブ化する前に、ユーザーに新しいアプリケーションでテストしてもらうことをお勧めします。以下の手順に従ってください。

    1. 古いJira / Confluenceアプリを非表示にするには：

      • 旧Jira/Confluenceアプリに進み、［General（一般）］タブを選択します。

      • [アプリケーションのアイコンをユーザーに表示しない]に表示する。

      • アプリがモバイルアクセス用にセットアップされている場合、［Do not display application icon in the okta mobile app（アプリケーションのアイコンをOktaモバイルアプリに表示しない）］チェックボックスもオンにします。

      • ［Save（保存）］をクリックします。

        • これで、旧Jira/Confluenceアプリがエンドユーザーに非表示になりました。

      +

    2. 古いJira / Confluence Cloudアプリケーションを非アクティブ化するには：

      • Jira / Confluence Cloudアプリケーションラベルの下にある アクティブ ステータスドロップダウンメニューをクリックし、 非アクティブ化：をクリックします

    3. 古いJira / Confluence Cloudアプリケーションを削除するには：

      • 上記の手順bの説明に従って、古いアプリを非アクティブ化します。アプリが非アクティブ化された後、 非アクティブ ステータスドロップダウンで。アプリを アクティブ化 または 削除 するオプションがあります。 削除を選択します。アプリケーションを本当に削除するかどうかを確認するように求められます。アプリケーションを削除をクリックする。
