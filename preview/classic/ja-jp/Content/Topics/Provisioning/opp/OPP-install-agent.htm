Install the Okta Provisioning Agent

Before configuring on-premises provisioning for an app, install the Linux or Windows Okta Provisioning Agent. Okta Provisioning Agentを複数のオンプレミス・アプリに接続できますが、アプリごとに一意のSCIMサーバーのURLを指定する必要があります。

Linux

  1. Enable the Transport Layer Security 1.2 protocol.
  2. (missing or bad snippet) ] []をダウンロードし、[ Okta Provisioning Agent（x64 RPM）の最新をダウンロード] をクリックします。
    または、アプリ・インスタンス・ページで ] [プロビジョニング ] タブに移動し、[] [プロビジョニング・エージェントをダウンロード] をクリックします。
  3. 任意。Okta Provisioning Agentファイルの整合性の検証に役立つように、ファイル・ハッシュ（SHA-512）を生成してファイル・サイズを確認します。
    1. Okta Provisioning AgentをダウンロードしたLinuxサーバーにサインインし、次のコマンドを使用してOkta Provisioning Agentのファイルハッシュ（SHA-512）を生成します。

    sha512sum setup.rpm

    setup.rpm を、手順1でダウンロードしたOkta Provisioning Agentのファイル・パスに置き換えます。

    1. Okta Provisioning Agentのファイル・サイズを確認するには、次のコマンドを使用します。

    ls -l setup.rpm

    Replace setup.rpm with the file path of the Okta Provisioning Agent you downloaded in step 1.

    1. Okta管理コンソールで、 [設定]> [ダウンロード]]に移動します。
    2. [Okta Provisioning Agents]領域まで下にスクロールし、手順aおよびbで取得したファイル・ハッシュとファイル・サイズの情報を、Okta Provisioning Agent（x64 RPM）で表示された情報と比較します。
  4. Okta Provisioning Agentがダウンロードされたら、rootとしてLinuxサーバーにサインインします。
  5. Copy the Okta Provisioning Agent .rpm file to a scratch directory, and then cd to that directory.
  6. 次のように入力して、yumを使用してインストールします。

    7. yum localinstall<package name>

    For example, yum localinstall OktaProvisioningAgent*.rpm

  7. 続行するように求められたら、 yと入力します。
  8. インストールが成功したら、画面にコマンドをコピーして、rootとしてスクリプトを実行します。

    9. sudo /opt/OktaProvisioningAgent/configure_agent.sh

Okta Provisioning Agentエージェントがインストールされると、OktaProvisioningAgentプロセス識別番号（.pid）ファイルがこのパス/ var / runに作成され、/ var / run / OktaProvisioningAgent /には作成されません。

  1. プロンプトで組織のURLを入力します（例：https://mycompany.okta.com）。
  2. ブラウザで指定したURLにアクセスし、ユーザー名とパスワードを使用してサインインします。
  3. Okta ProvisioningAgentがOkta APIにアクセスできるようにするには、[] [アクセスを許可] をクリックします。

    4. 注：TLS 1.2プロトコルを有効にしていない場合、または以前のバージョンを使用している場合は、 Enable the Transport Layer Security 1.2 protocol]を参照してください。

  4. Return to the command line. 構成が正常に完了したというメッセージを受け取ったら、次のコマンドをコピーして入力します。

    5. service OktaProvisioningAgent start

  5. To confirm that the Okta Provisioning Agent is running, enter the following:

    6. service OktaProvisioningAgent status

Windows

  1. Enable the Transport Layer Security 1.2 protocol.
  2. (missing or bad snippet) Downloads.
  3. Windows Okta Provisioning Agentの [最新をダウンロード] をクリックします。
  4. 任意。ファイルハッシュ（SHA-512）を生成し、ファイルサイズを確認して、Okta Provisioning Agentファイルの整合性を検証します。
    1. コマンド・プロンプトを開き、次のコマンドを使用して、Okta Provisioning Agentのファイル・ハッシュ（SHA-512）を生成します。

    CertTool -hashfile setup.exe SHA512

    setup.exe を、手順2でダウンロードしたOkta Provisioning Agentのファイル・パスに置き換えます。

    1. Okta Provisioning Agentをダウンロードした場所を参照してファイルを右クリックし、 Propertiesを選択し、 Size フィールドに表示された値を記録して、 OKをクリックします。
    2. On the Okta Admin Console, go to Settings > Downloads.
    3. [Okta Provisioning Agent]領域まで下にスクロールし、手順aおよびbで取得したファイル・ハッシュとファイル・サイズの情報を、Okta Provisioning Agent（Windows x64 EXE）で表示された情報と比較します。
  1. インストーラーを起動し、[次へ] をクリックします。
  2. [ライセンス契約]ダイアログボックスで、[次へ]をクリックします。
  3. 任意。インストールフォルダを変更し、 ［インストール]をクリックします。
  4. Okta Customer DomainのURLを入力し、[次へ] をクリックします。

When the Okta Provisioning Agent agent is installed, the OktaProvisioningAgent process identification number (.pid) file is created at this path /var/run and not /var/run/OktaProvisioningAgent/.

  1. In your browser, sign in to your org.
  2. Grant permission to access the Okta API by clicking Allow Access.
  3. インストーラーに戻り、［完了］をクリックします。
  4. Oktaにサインインします。

    5. Note: If you haven't enabled TLS 1.2 protocol or are using an earlier version, see Enable the Transport Layer Security 1.2 protocol.

  5. 管理コンソールで、[エージェント]を選択します。構成済みのOkta Provisioning Agentがリストに含まれていることを確認します。
