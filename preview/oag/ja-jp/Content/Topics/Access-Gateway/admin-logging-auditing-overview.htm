ログの形式と例

Access Gatewayは監査のためにすべてのイベントとアクションをログに記録します。これには管理者のアクションとユーザーのアクセス、認証状態が含まれます。

項目

  • 一般的なログの形式
  • Admin UIコンソール
  • ゲートウェイ
  • プロセスモニター

一般的なログの形式

Access Gatewayは監査イベントを以下の形式でログに記録します：

TIMESTAMP HOSTNAME APPLICATION PROCID COMPONENT SUB-COMPONENT LOG_LEVEL EVENT [STRUCTURED_DATA] MESSAGE

ログステートメントのフィールド

フィールド

説明

TIMESTAMP

現在のシステム日付と時刻

HOSTNAME

マシンのホスト名

APPLICATION

Access Gateway

PROC_ID

プロセス層

COMPONENT

プロセスのコンポーネント

SUB-COMPONENT

プロセスのサブコンポーネント

LOG_LEVEL

ログレベル

EVENT

イベントの種類

STRUCTURED_DATA

分析／トラブルシューティングに重要なイベント発生に関するデータ

MESSAGE

読み取り可能なメッセージ

システムIDPステータス

ログ識別子

フィールド 値

PROC_ID

WEB_CONSOLE

COMPONENT

IDP

SUB-COMPONENT

LOCAL

EVENT

SYSTEM_IDP_STATUS

構造化データ

フィールド 説明

NAME

IDP名

DOMAIN

IDPドメイン

TYPE

IDPタイプ

RESULT

PASS/FAIL

REASON

VALID / INVALID_NETWORK_CONN (FAIL)、INVALID_TOKEN (FAIL)

  1. 有効化なIDP

    1. ログレベル：INFO

    2. RESULT: PASS

    3. REASON: VALID

    4. メッセージ：Success confirming IDP status with: <IDP Domain>（IDPステータスの確認に成功: ＜IDPドメイン＞）

    5. ログの例：

      Oct 9 04:00:00 Access Gateway WEB_CONSOLE IDP LOCAL INFO SYSTEM_IDP_STATUS [NAME="<IDP Name> IDP" DOMAIN="<IDP URL>" TYPE="<Identity Provider type>" RESULT="PASS" REASON="VALID"] Success confirming IDP status with: <IDP URL>

  2. IDPはネットワーク到達不可能

    1. ログレベル：ALERT

    2. RESULT: FAIL

    3. REASON: INVALID_NETWORK_CONN

    4. メッセージ：Failure confirming connectivity with IDP: <IDP Domain>. Please verify your network configuration.（IDP：＜IDPドメイン＞を使用した接続の確立に失敗しました。ネットワークコンフィギュレーションを確認してください）

    5. ログの例：

      Oct 9 04:02:00 Access Gateway WEB_CONSOLE IDP LOCAL INFO SYSTEM_IDP_STATUS [NAME="<IDP Name> IDP" DOMAIN="<IDP URL>" TYPE="<Identity Provider type>" RESULT="FAIL" REASON="INVALID_NETWORK_CONN"] Failure confirming connectivity with IDP: <IDP URL>>. Please verify your network configuration.

  3. IDPセキュリティ鍵が有効でなくなった

    1. ログレベル：ALERT

    2. RESULT: FAIL

    3. REASON: INVALID_TOKEN

    4. メッセージ：Failure validating security token with IDP: <IDP Domain>. Please validate token exists and is enabled.（セキュリティトークンのIDP：＜IDPドメイン＞との検証に失敗しました。トークンが存在していて、有効であることを確認してください）

    5. ログの例：

      Oct 9 04:02:23 Access Gateway WEB_CONSOLE IDP LOCAL INFO SYSTEM_IDP_STATUS [NAME="<IDP Name> IDP" DOMAIN="<IDP URL>" TYPE="<Identity Provider type>" RESULT="FAIL" REASON="INVALID_NETWORK_CONN"] Failure validating security token with IDP: <IDP Domain>. Please validate token exists and is enabled.

    1. ログレベル：INFO

    2. メッセージ：User login:<Username>（ユーザーログイン：＜ユーザー名＞）

  2. 時間が同期していない

    1. Log Level: ERROR

    2. RESULT: FAIL

    3. REASON: Invalid SAML Assertion（無効なSAMLアサーション）

このセクションでは、セッション関連の問題のトラブルシュートのために監査ログを使用して追跡可能な通常の認証フローを説明します。各ユーザーセッションは一意なセッションIDに割り当てられています。このセッションIDはユーザーセッションの追跡にも使われ、トラブルシュートやデバッグにも役立ちます。

基本的な認証とセッション作成のフローを、生成された監査ログのシーケンスと共に説明します：

  1. ブラウザーがAccess Gatewayにアプリケーションへのアクセスのリクエストを送信します。Access Gatewayがセッションが既に存在していることを確認し、ブラウザーを認証のためにIDPにリダイレクトします。
