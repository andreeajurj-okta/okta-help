Okta多要素認証を使用してOffice 365向けのAzure AD多要素認証の要件を満たす

This is an Early Access feature. To enable it, use the Early Access Feature Manager as described in 早期アクセス機能とベータ機能の管理.

Okta多要素認証を使用して、WS-Federation Office 365アプリ・インスタンス向けのAzure AD多要素認証の要件を満たすことができます。

Okta多要素認証は次のユース・ケースで使用できます。

  • Oktaとフェデレーションされたドメインの、Azure AD条件付きアクセスによってトリガーされる多要素認証プロンプトの多要素認証要件を、Oktaで処理する必要がある。
  • OktaとMicrosoft両方の多要素認証のニーズに対応する単一の多要素認証ソリューションを使用できるように、Okta多要素認証を使用してエンド・ユーザーをWindows Hello for Businessに登録することができる。

既知の問題

ユーザーが無限サインイン・ループに陥ることがある

Oktaアプリ・レベルのサインオン・ポリシーが、Azure ADポリシーよりも弱い場合、エンド・ユーザーが無限サインイン・ループに陥ることがあります。 これは、次のシナリオで発生する可能性があります。

  • アプリ・レベルのサインオン・ポリシーで多要素認証が必要とされない

    この場合、Oktaではアプリにアクセスする際に多要素認証が求められません。しかし、Azure AD条件付きアクセスには多要素認証が必要であり、入力された多要素認証クレームがOktaによって渡されることが期待されます。その結果、ユーザーが無限認証ループに陥ります。組織レベルで構成されている場合にもOktaによって多要素認証を求められることがありますが、この多要素認証クレームはAzure ADに渡されません。 

  • アプリ・レベルのサインオン・ポリシーでは、ユーザーが「ゾーン内」のネットワークからサインインする場合は多要素認証は必要ありませんが、ユーザーが「ゾーン外」のネットワークからサインインする場合は多要素認証が必要です。

    この場合、ユーザーがゾーン内のネットワークからサインインしても、多要素認証は求められません。しかし、Azure AD条件付きアクセスには多要素認証が必要であり、入力された多要素認証クレームがOktaによって渡されることが期待されます。その結果、ユーザーが無限認証ループに陥ります。

開始する前に

  1. Oktaで多要素認証を構成する：「アプリのサインオン・ポリシー」の説明に従って、WS-Federation Office 365アプリ・インスタンス向けのアプリのサインオン・ポリシーを構成します。

  2. Azure ADで多要素認証を構成する：Microsoftドキュメンテーションの説明に従って、Azure ADインスタンスで多要素認証を構成します。

この手順を開始する

Okta多要素認証のサポートを有効にするには、Office 365ドメインのフェデレーション設定を変更する必要があります。ドメインのフェデレーションを手動で行ったか自動で行ったかに応じて、次の手順のいずれかを選択します。

手動でフェデレーションされたドメインの場合

設定手順から更新されたフェデレーション・スクリプトを実行します。

  1. Okta管理コンソールから、[アプリケーション] > [アプリケーション]に移動します。
  2. WS-Federationが行われたOffice 365アプリを開きます。

  3. [サインオン]タブ > [サインオン方法] > [WS-Federation] > [設定手順を表示]をクリックします。

    [Office 365 WS-Federationの設定方法]ページが開きます。

  4. このページで、[ドメインがすでにフェデレーションされている場合]セクションに移動します。
  5. このセクションからスクリプトをコピーして、Windows PowerShellで実行します。

  6. 次のPowerShellコマンドを実行して、SupportsMfaの値がTrueであることを確認します。

    Connect-MsolService

    Get-MsolDomainFederationSettings -DomainName <yourDomainName>

    結果の例

    ActiveLogOnUri : https://example.okta.com/app/office365/issueruri/sso/wsfed/active

    DefaultInteractiveAuthenticationMethod :

    FederationBrandName : Okta

    IssuerUri : issueruri

    LogOffUri : https://example.okta.com/app/office365/issueruri/sso/wsfed/signout

    MetadataExchangeUri : https://example.okta.com/app/office365/issueruri/sso/wsfed/mex

    NextSigningCertificate :

    OpenIdConnectDiscoveryEndpoint :

    PassiveLogOnUri : https://example.okta.com/app/office365/issueruri/sso/wsfed/passive

    SigningCertificate : <SigningCertificate>

    SupportsMfa : True

自動的にフェデレーションされたドメインの場合

この場合、設定を構成する必要はありません。以下に説明されているように、[編集]をクリックしてから[保存]をクリックするだけです。

  1. Okta管理コンソールから、[アプリケーション] > [アプリケーション]に移動します。
  2. WS-Federationが行われたOffice 365アプリを開きます。
  3. [サインオン]タブ > [編集] > [保存]をクリックします。

  4. 次のPowerShellコマンドを実行して、SupportsMfaの値がTrueであることを確認します。

    Connect-MsolService

    Get-MsolDomainFederationSettings -DomainName <yourDomainName>

    結果の例

    ActiveLogOnUri : https://example.okta.com/app/office365/issueruri/sso/wsfed/active

    DefaultInteractiveAuthenticationMethod :

    FederationBrandName : Okta

    IssuerUri : issueruri

    LogOffUri : https://example.okta.com/app/office365/issueruri/sso/wsfed/signout

    MetadataExchangeUri : https://example.okta.com/app/office365/issueruri/sso/wsfed/mex

    NextSigningCertificate :

    OpenIdConnectDiscoveryEndpoint :

    PassiveLogOnUri : https://example.okta.com/app/office365/issueruri/sso/wsfed/passive

    SigningCertificate : <SigningCertificate>

    SupportsMfa : True

この機能を無効にする

この機能をオフにする場合は、Oktaで自動的にフェデレーションが行われ、この機能が有効になっているすべてのフェデレーション・ドメインについて、SupportsMfa設定を手動でfalseに設定する必要があります。

このPowerShellコマンドレットを使用してこの機能をオフにします。

Set-MsolDomainFederationSettings -DomainName <targetDomainName> -SupportsMfa $false

機能の仕組み

Okta多要素認証がAzure AD条件付きアクセス多要素認証の要件を満たす

Office 365が多要素認証を必要とするAzure AD条件付きアクセス・ポリシーで構成されている場合、アプリにアクセスを試みるエンド・ユーザーは、Azure AD多要素認証の要件を満たすために、Oktaによって多要素認証を求められます。次に、Oktaによって成功した多要素認証クレームがAzure ADに渡されます。Azure ADによってこのクレームが受け入れられ、エンド・ユーザーに個別の多要素認証が求められることなく、アクセスが許可されます。

次の表に示されているように、Azure AD条件付きアクセス多要素認証が有効になっていて、アプリ・レベルでOkta多要素認証が有効になっている場合に、Oktaによって多要素認証クレームが渡されます。

Oktaアプリ・レベル多要素認証Azure AD多要素認証実行される操作の内容
無効有効

エンド・ユーザーが無限サインイン・ループに陥ります。これを防ぐには、Azure AD多要素認証の要件を満たすためにOkta多要素認証を構成する必要があります。

有効有効エンド・ユーザーはOktaで多要素認証プロンプトに入力します。入力された多要素認証クレームがOktaによってAzure ADに渡されます。Azure ADによってOktaからの多要素認証が受け入れられ、個別の多要素認証は求められません。ユーザーはOffice 365へのアクセスを許可されます。

OktaによってユーザーがWindows Hello for Businessに登録される

前提条件：デバイスがハイブリッドAzure ADまたはAzure ADに参加している必要があります。

組織でWindows Hello for Businessが必要な場合、Windows Hello for Businessにまだ登録されていないエンド・ユーザーは、Oktaでステップアップ認証（SMS、プッシュなど）を入力するよう求められます。Windows Hello for Businessへの登録に成功したら、エンド・ユーザーはそれを使用してデバイスにログインできます。次の表に示されているように、OktaではWindows Hello for Businessへのエンド・ユーザーの登録が支援されます。

Oktaアプリ・レベル多要素認証実行される操作の内容
無効

エンド・ユーザーが無限サインイン・ループに陥ります。これを防ぐには、Azure AD多要素認証の要件を満たすためにOkta多要素認証を構成する必要があります。

有効エンド・ユーザーはOktaでステップアップ多要素認証プロンプトに入力します。Windows Hello for Businessへの登録に成功すると、エンド・ユーザーはAzure AD多要素認証を満たすための要素としてWindows Hello for Businessを使用できます。
