ゲートウェイの高可用性

これはEarly Accessの機能です。有効にする場合は、Oktaサポートにお問い合わせください。

ゲートウェイはさまざまな手法を使用して高可用性を確保し、従来のSSH要塞よりも高い信頼性を提供します。

項目

  • ゲートウェイのトラブルシューティング

ゲートウェイの負荷分散

プロジェクトで使用できるゲートウェイが複数存在する場合、クライアントがサーバーにアクセスしようとするたびに、ゲートウェイの１つがランダムに選択されます。これには、プロジェクトに有効なすべてのゲートウェイ全体で自動的に負荷分散するリクエストを提供する効果を持ちます。１つのゲートウェイに障害が発生すると、他のゲートウェイが引き続きリクエストの処理を継続できます。ただし、追加の構成がないと、ダウンしたゲートウェイに送信されたリクエストは失敗します。

リクエストが適切に処理されるのを確実にするには、ロードバランサーの背後にあるWebアプリケーションサーバーを構成する方法と同様に、ゲートウェイの負荷分散を構成する必要があります。標準の負荷分散を次のように構成します。

  1. サーバーがポート7234のロードバランサーまたはユーザーが構成した別のポートのロードバランサーに接続できることを確認します。
  2. ロードバランサーがポート7234またはユーザーが構成した別のポートのすべてのゲートウェイに接続できることを確認します。
  3. ゲートウェイがポート22のSSHを使用してターゲットサーバーに接続できることを確認します。
  4. すべてのゲートウェイが同じAccessAddressを使用するように構成します。このアドレスは、ロードバランサーへのアクセスに使用される、ドメイン名または静的 IPのアドレスにしてください。異なるポートは異なるアドレスとして扱われます。
    • クライアントがゲートウェイへの接続方法を知る必要があることに加え、クライアントに対してゲートウェイの識別子を証明するために使われるホスト証明書が有効であると見なされるよう、この設定が正確である必要があります。
    • 複数のゲートウェイのホスト証明書は、同一のデフォルトアドレスを共有する場合に有効であると見なされ、この有効性は、Amazon Web Services（AWS）またはGoogle Cloud Platform（GCP）を使用する場合、クラウドインスタンスメタデータを使用して検出されます。

ロードバランサーの背後のゲートウェイを設定した後、ゲートウェイの機能性に問題があったり到達不可能な場合にロードバランサープールからゲートウェイを削除する余分なヘルスチェックを追加することをお勧めします。適正なヘルスチェックの例は、次のとおりです。

  • ゲートウェイが正しいポート上（デフォルトでは7234ポート）でリッスンしていることを確認します。
  • ゲートウェイに十分なストレージがあることを確認し、ローカルストレージを使用している場合はログを保存するのに十分であるか、またはクラウドストレージの場合はストレージに移動する前にログを一時的に保存するのに十分であるかを確認します。
  • ゲートウェイのメモリおよびCPU使用率が過剰でないことを確認します。

使用するプラットフォームとツールに対してヘルスチェックを実装する方法に関する詳細は、クラウドプロバイダーとロードバランサーのドキュメントをご覧ください。

ゲートウェイのステータスチェック

ゲートウェイはヘルスに関するデータをアドバンストサーバーアクセスに自動的に送信するので、ユーザーはこれを使用してゲートウェイを管理することができます。ゲートウェイの最新のヘルスデータの詳細は、アドバンストサーバーアクセスのダッシュボードで確認できます。

ゲートウェイのステータスチェックは、ゲートウェイの管理に役立つだけでなく、ユーザーがゲートウェイを使用してサーバーへのアクセスを試行すると、特定のゲートウェイが選択されているかどうかを制御できます。

ゲートウェイはステータスを2分毎に報告します。ゲートウェイがステータスを報告してから5分間以上経過すると、プールから自動的に削除されます。いずれかのゲートウェイがステータスを報告してから5分間以上経過すると、その接続は過去24 時間以内にステータスを最後に報告したゲートウェイに送信されます。過去24 時間以内にステータスを報告したゲートウェイが存在しない場合、その接続は失敗します。

ゲートウェイを使用せずにサーバーへの一時的なアクセスを作成する

不測の事態が生じ、高可用性をサポートするために使用されるすべてのメカニズムに障害が発生し、ゲートウェイを経由せずにサーバーへの制限付きアクセスを許可する必要がある場合、次の手順を実行して一時的な監査付きサーバーアクセスを許可することが可能です。

  1. 一時的なアクセスを許可する必要があるユーザーのみを含んだ一時的なグループを作成します。
  2. アクセスするサーバーが登録されているプロジェクトから、すべてのグループを削除します。
  3. 一時的なグループをプロジェクトに追加します。

新しい権限承認構成がサーバーに同期されます。このプロセスには数分かかり、その後一時グループのメンバーはサーバーにアクセスできるようになります。インシデントの解決後、古い構成を復元することができます。

注意： この方法を使用してサーバに作成された接続は、ゲートウェイを経由しないので、これらのセッションはログに記録されません。ただし、サーバーに誰が接続したか、いつ接続されたかの情報は、アドバンストサーバーアクセスのイベントログに記録されます。

ゲートウェイのトラブルシューティング

権限を持つユーザーがSSHによりゲートウェイに接続してトラブルシューティングを実行できるよう、アドバンストサーバーアクセスを構成することができます。これを行うには、アドバンストサーバーアクセスのサーバーエージェントを使用して、ゲートウェイへのアクセスを管理し、ゲートウェイを必要としないプロジェクトにもゲートウェイを登録します。サーバーがアクティブかつアクセス可能な場合、ゲートウェイプロジェクトに属するユーザーは、ゲートウェイを備えるサーバーにSSHを使用して接続することができます。
