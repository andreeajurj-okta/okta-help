General Page

Use the General page to set general security policies for your organization and specific security policies for mobile usersIn Okta literature, we generally refer to "users" as the people who serve as Okta administrators. When we refer to "end users" we are generally referring to the people who the administrators serve. That is, those who use Okta chiclets to access their apps, but have no administrative control.. If you have Group Password Policy enabled, use the Password section to enforce the use of strong passwords to better protect your organization's assets.

Password

The password policy settings enable you to define and enforce the use of strong passwords to better protect your organization's assets. See Configuring an Organization-Wide Password Policy, below, for details.

Organization

To configure the following organization security policies, click the Edit button.

"Remember me" checkbox on login page: Enable this policy to store the username in a cookie on a browser. The cookie only remembers the last user who successfully signed in. This policy does not remember your password. Important: Applications that use embedded browsers cannot remember the user name, as they do not store persistent cookies. These applications include thick clients such as Outlook and native applications on mobile devices..

Activation emails are valid for: When you add new users, they are sent emails that enable them to activate their accounts. Use this feature to limit the amount of time activation emails are valid.

Okta Mobile

To configure the following organization security policies, click the Edit button.

Prompt for PIN when user is inactive for: Select the amount of time that can pass before users are automatically signed out and must reenter their PINs to sign back in.

PIN expires after: Select the amount of time before your users' PINs expire and they must select a new one.

Configuring Additional Organization-Wide User Account Settings

Note: This section describes how to configure some orgAn abbreviation of organization, but can also be thought of as a company. A company that uses Okta as their SSO portal is generally referred to as an org. As an administrator, you decide how Okta should be displayed and/or integrated with your org.-wide settings in the current Generally Available version of the platform. If the Early Access (EA) feature Group Password Policies is enabled for your org, settings marked with an asterisk (*) below are located in Security > Policies (see Creating Group Password Policies). Additionally, unless the EA feature Group Password Policies is enabled for your org, password policy settings are configured by Okta, cannot be changed, and do not apply to Active Directory-mastered users.

Okta has configured the password reset flow such that when an end user resets their password (either through the Okta UI or the Okta API), it cannot be reset to their existing password. This element is not configurable.

You can configure the following org-wide settings:

  • Session lifetime — Configure this policy to set a time-out duration for a session. If a user's session exceeds this time, the user is required to sign back into the appAn abbreviation of application. Essentially, it is a web-based site used to perform any number of specific tasks, and requires authentication from end users by signing in.. Session lifetime can be set from one minute to 12 hours.
  • "Remember me" checkbox on sign in page — Select this option if you want to store the username in a cookie on a browser. The cookie only remembers the last user who successfully signed in. Note that this setting does not remember your password.
  • *Show lockout failures — When this option is enabled, locked-out users are shown a message when they try to sign-in that they are locked out. When this setting is disabled, locked-out users are shown a message stating only that their sign-in attempt has failed. Okta resets the lockout time of Active Directory-mastered users to zero after they complete the password reset procedure in AD. This enables these users to sign back in.
  • *Allow SMS for self-service operations (such as reset password or unlock account) — Select if you want to allow end usersIn Okta literature, we generally refer to "end users" as the people who have their own Okta home page (My Applications), using chiclets to authenticate into all of their apps. End users do not have any administrative control. When we refer to "users" we are generally referring to the individual(s) who have administrative control. who have forgotten their password to be able to reset passwords using SMS. If end users forget their passwords, Okta sends them a text message with a password reset code. The first time end users log on after this feature is enabled, they have the option to either enter a phone number, or click Remind me later to receive a reminder in 30 days.
  • Note: This feature is free for customers in the U.S. and Canada. Admins with international orgs should contact customer support before enabling SMS.
  • Activation emails are valid for — Specify how long new user account activation emails are valid.
