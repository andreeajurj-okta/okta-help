Release: 2021.04.2

Deployment date: April 28, 2021

Features and enhancements

Email templates

The text in the magic link email sent to end users has been clarified.

Multifactor Authentication

On the MFA Phone Enrollment screen, messages and button labels now indicate whether the user will receive the verification code by SMS or voice call.

On MFA enrollment setup and password reset screens, styling and messaging has been improved.

In orgs with the WebAuthn authentication method enrolled and set to User Verification Required, users who authenticate with WebAuthn are now always challenged to provide user verification (PIN or biometrics). This update ensures compliance with NIST AAL2 requirements.

Okta Verify

End users are now guided through the remediation steps if the biometrics settings in Okta Verify are no longer valid.

Existing Okta Verify users no longer need to provide the username when they are prompted to enable or reset biometrics.

Sign-In Widget

A context banner now displays the app name or logo when a user attempts to sign in directly to an app. See Customize text on your sign-in page.

For details about this release, see the Okta Sign-In Widget Release Notes. For more information about the Widget, see the Okta Sign-In Widget Guide.

System Log

When a user authenticates with Okta FastPass, the System Log now displays details in the Device event. This information may be useful for troubleshooting and other analysis. Admins can look up the ID in Directory > Devices for more information about the device.

Release: 2021.04.0

Deployment date: March 31, 2021

Features and enhancements

Admin Experience Redesign

The Okta Admin Console now has:

  • A modern look and feel with improved responsiveness for the new navigation side bar.

  • A redesigned Okta Admin Dashboard that displays more practical insights for admins.

  • An Agents page in the Admin Dashboard that shows the status and version of every Okta agent that is connected to customers' on-premises servers.

This improves the accessibility of the product, improves admin productivity, and helps admins to be more proactive with security issues.

See Admin Experience Redesign.

Email templates

Okta Identity Engine now supports customization of the following templates:

  • Authenticator Enroll: Sends users a confirmation email if they enroll in a new authenticator for their account.

  • Authenticator Reset: Sends users an email if they or an admin resets an authenticator for their account.

See Custom email and SMS templates.

The Email Link and Email Verification templates now contain a token expiry message for users.

Multifactor Authentication

On the Authenticators Setup page and Enrollment page, the Used for section indicates if an authenticator is used for authentication, password reset, or both.

Policies

For app sign-on policy rules, the option AND Okta Verify user interaction that requires users to prove their presence is updated for clarity. The option is now Access with Okta FastPass is granted.

See Add an app sign-on policy rule.

Sign-in Widget

Sign-in Widget v5.5.0 now supports:

  • Registration hooks: click, parseSchema, preSubmit, and postSubmit.

  • The Show Password toggle switch allows users to see their password as they type.

  • Configuration of the transformUsername function to customize the username that is sent to Okta.

  • Sign-In Widget restyling to remove blank space if you choose to remove the security image.

  • The Remember me check box label now reads Keep me signed in.

See Configure a custom Okta-hosted sign-in page.

System Log

You can now filter using new Advanced Filters operators:

  • ends with

  • not equal

  • is present (value exists)

  • greater than

  • greater than or equal to

  • less than

  • less than or equal to

Additionally, you can now use the not equal, ends with, and is present operators in the System Log search bar. These operators provide greater flexibility when filtering System Log events.

See System Log filters and search.

ThreatInsight

Orgs that use free trial editions now see a limited functionality notification in the Okta ThreatInsight Settings section of the Security > General page.

See General Security.

Okta Verify

Beginning with version 2.0.1, you can integrate Windows Okta Verify with some Endpoint Detection and Response (EDR) solutions.

When users try to access a protected resource, Okta Verify probes their device for context and trust signals and uses these to inform the access decision. EDR integration extends device posture evaluation by enabling Okta Verify to capture signals collected by your EDR client running on the same device. Okta currently integrates with CrowdStrike ZTA and Microsoft Windows Security Center.

See Integrate Okta Verify with Endpoint Detection and Response solutions.

Release: 2021.02.3

Deployment date: February 24, 2021

Features and enhancements

Self-Service Password Reset

Previously, email and phone (voice call, SMS) were the only available authenticators for self-service password reset. Now in Identity Engine, we're introducing more flexible options.

In addition to using email, voice call, and SMS, users can now reset their password using a stronger authenticator like Okta Verify Push. If an admin requires a step-up authentication, end users can use any enrolled authenticator.

Once configured in a password policy rule, self-service password reset offers an improved password recovery experience with a strengthened security posture.

To learn how to configure this feature, see Self-Service Password Reset.

Release: 2021.02.0

Deployment date: February 3, 2021

Features and enhancements

Active Directory Delegated Authentication

Identity Engine now supports Active Directory (AD) delegated authentication. When delegated authentication is enabled, AD user data is used to authenticate users. Delegated authentication simplifies user management by making AD the single source of truth for user data and access credentials.

See About delegated authentication with Active Directory.

Authorization

Validation error messaging for OAuth Authorization and Interaction Code token exchanges is now more descriptive and corrects some faulty messaging.

First Party Applications

Okta admins can now create app-based sign-on policies for the Okta Dashboard, Okta Admin Console, and Okta Browser Plugin.

Previously, sign-on policies couldn't be configured for these first party applications. With this release, policy based on context such as user location, device, behavior, risk level, group membership, and more is included. This gives admins more flexibility and granular control over sign-on requirements for these first party apps. For example, different MFA requirements might apply to the Okta Admin Console for different groups of people. See Create sign-on policies with Okta Applications

Sign-on policies

The org-wide limit for app sign-on policies has increased. Admins can now configure unique sign-on policies for 5,000 different applications.

See App sign-on policies.
