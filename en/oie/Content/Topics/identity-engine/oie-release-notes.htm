Identity Engine release notes

Limited GA: Okta Identity Engine is currently available only to selected customers.

Release: 2021.04.0

Deployment date: March 31, 2021

Features and enhancements

  • Admins can customize the following email templates in Identity Engine:

    • Authenticator Enroll sends end users a confirmation email if they enroll in a new authenticator for their account.

    • Authenticator Reset sends end users an email if they or an admin resets an authenticator for their account.

    See Custom email and SMS templates.

  • On the Authenticators Setup page and Enrollment page, the Used for section indicates if an authenticator is used for authentication, password reset, or both.

  • Sign-in Widget version 5.5.0 contains these enhancements.

    • The Sign-In Widget now supports these Registration hooks: click, parseSchema, preSubmit, and postSubmit.

    • The Show Password toggle switch allows users to see their password as they type.

    • Admins can now configure a transformUsername function to customize the username that is sent to Okta.

    • Admins can now restyle the Sign-In Widget to remove blank space if they removed the security image.

    • The Remember me check box label now reads Keep me signed in.

    See Configure a custom Okta-hosted sign-in page.

  • For app sign-on policy rules, the option AND Okta Verify user interaction that requires users to prove their presence is updated for clarity. The option is now Access with Okta FastPass is granted. See Add an app sign-on policy rule.
  • The Email Link and Email Verification templates now contain a token expiry message for users.
  • Orgs that use free trial editions now see a limited functionality notification in the Okta ThreatInsight Settings section of the Security > General page. See General Security.

  • With the Admin Experience Redesign feature, the Okta Admin Console now has:

    • A modern look and feel with improved responsiveness for the new navigation side bar.

    • A redesigned Okta Admin Dashboard that displays more practical insights for admins.

    • An Agents page in the Admin Dashboard that shows the status and version of every Okta agent that is connected to customers' on-premises servers.

    This improves the accessibility of the product, improves admin productivity, and helps admins to be more proactive with security issues. See Admin Experience Redesign.

  • Admins can now filter using new Advanced Filters operators:

    • ends with

    • not equal

    • is present (value exists)

    • greater than

    • greater than or equal to

    • less than

    • less than or equal to

    Additionally, admins can now use the not equal, ends with, and is present operators in the System Log search bar. These operators provide greater flexibility when filtering System Log events. See System Log filters and search.

    • Scrolling functionality didn’t work as expected on some pages.

    • The Okta Admin Dashboard reached the rate limit threshold rapidly, causing a failure to load data in the Admin Dashboard widgets.

    • The spotlight search input field had extra padding.

    • Some pages had layout issues.

    • Some dialog boxes had unwanted scrollbars.

    • Some conditions in group rules were unreadable.

    • Group icons weren’t displayed properly on the Group Assignment page.

      • (OKTA-337030, OKTA-378809, OKTA-380636, OKTA-381639)

      (OKTA-375978, OKTA-381076, OKTA-380069)

  • Some users weren’t able to use primary email for self-service password reset when the email address differed from the username. (OKTA-374364)
  • If an admin attempted to add a desktop platform in Security > Device Management by generating a SCEP URL but then changed their selection to Use my own certificate authority and saved that setting, two desktop platforms were added instead of one. (OKTA-344793)
  • After orgs migrated to Identity Engine, some users were locked out of applications because of policies created in classic mode. New applications with a “deny all” rule in their sign-on policy did not display a locked icon. (OKTA-361006)
  • The Actions drop-down lists didn’t work correctly on the Authenticators > Setup page. (OKTA-377680)
  • When Self-Service Password Reset was enabled, end users could change their LDAP password from the End-User Dashboard settings even if the policy didn't allow it. (OKTA-374967)
  • On the Okta Sign-on Policy page, the Remember Device by Default check box appeared but was unsupported. (OKTA-378912)

  • Email templates for Registration-Activation and Registration-Email Verification were untranslated. (OKTA-380697)

Release: 2021.03.3

Deployment date: March 24, 2021

Features and enhancements

  • Beginning with version 2.0.1, you can integrate Windows Okta Verify with some Endpoint Detection and Response (EDR) solutions.

    When users try to access a protected resource, Okta Verify probes their device for context and trust signals and uses these to inform the access decision. EDR integration extends device posture evaluation by enabling Okta Verify to capture signals collected by your EDR client running on the same device. Okta currently integrates with CrowdStrike ZTA and Microsoft Windows Security Center.

    See Integrate Okta Verify with Endpoint Detection and Response solutions.

  • HealthInsight now shows recommendations for Identity Engine. HealthInsight audits an organization’s security settings and recommends tasks to improve the organization’s security posture. See HealthInsight.

  • Application integrations based on legacy SAML 1.1 templates are now supported.

  • The Authenticators Setup page and Authenticators Enrollment page now include a note that MFA enrollment policies can be affected by password policies.

  • In a self-hosted Sign-In Widget, admins can now configure a transformUsername function to customize the username that is sent to Okta. See Configure a custom Okta-hosted sign-in page.

  • Password enrollment for new users sometimes failed during self-service registration. (OKTA-373018)

Release: 2021.03.0

Deployment date: March 3, 2021

Features and enhancements

  • The Okta End-User Dashboard and Okta Browser Plugin have been redesigned with a modern look and feel that includes new sidebar navigation, fuzzy search, and sections that replaces tabs.

    • Admins can enable this new design all at once or by groups. The new experience is 50% faster, more intuitive to use, and more responsive to smaller screens. Design changes also improve accessibility and app discovery for end users. See Create sign-on policies with Okta Applications.

  • To help admins distinguish orgs that have upgraded to Okta Identity Engine, the Help menu in the Okta Admin Console now has a new Powered by Okta Identity Engine label.

  • The Sign-in Widget UI has been updated to improve the self-service password reset experience. See New user registration, activation, and sign-in experience.
  • Authenticators may not be removed or disabled if the default policy rule for self-service password reset doesn't specify a primary authenticator. See Self-service password reset.
  • To help prevent brute force attacks, rate limiting now applies when an end user authenticates with Okta Verify. If the user's attempt to sign in fails five times within five minutes, they will be denied authentication during that time frame.
  • When creating a sign-on policy, you can now create rules that apply only to LDAP Interface user authentications and exclude other authentication methods. See Sign-on policies and rules.
  • The text shown when Okta Verify is in edit mode has been updated for clarity and accuracy. The Usage section now states that Okta Verify can be used for both authentication and password recovery.

Bug fixes

  • When admins modified authenticators that were migrated from MFA factors in classic mode, disabled factor options were enabled. For example, after a phone factor without voice calls was migrated, unrelated updates to the phone authenticator enabled voice calls. (OKTA-369574)
  • With RADIUS enabled, enrolled users couldn’t verify with Okta Verify Push. After it was disabled, RADIUS continued to prompt for push verification. (OKTA-370533)
  • The Okta Sign-In Widget didn’t fully support internationalization and localization. (OKTA-362345)
  • When a custom sign-in page was configured, end users still saw a page with Okta branding. (OKTA-365678)
  • The MFA Usage Report didn't include the last authenticator enrollment date for an end user or any details on how they authenticated with Okta Verify. (OKTA-362254)
  • The System Log didn't specify the app sign-on policy rules that denied an end user from signing in to their account. (OKTA-365451)

Release: 2021.02.3

Deployment date: February 24, 2021

Features and enhancements

  • Self-Service Password Reset

    Previously, email and phone (voice call, SMS) were the only available authenticators for self-service password reset. Now in Identity Engine, we're introducing more flexible options.

    In addition to using email, voice call, and SMS, users can now reset their password using a stronger authenticator like Okta Verify Push. If an admin requires a step-up authentication, end users can use any enrolled authenticator.

    Once configured in a password policy rule, self-service password reset offers an improved password recovery experience with a strengthened security posture.

    To learn how to configure this feature, see Self-Service Password Reset.

Bug fixes

  • Users who attempted to sign in with an Okta account pending activation were prompted to enroll with a password. (OKTA-362727)

  • Internationalization and localization support for the Okta Sign-In Widget wasn't available for a number of self-service registration and sign-in related messages. (OKTA-362345)

  • The API endpoint /authorize, when called during the Identity engine implicit flow (with prompt=none) by an app with a policy rule violation, returned a 500 Internal Server Error rather than a valid error message. (API - OKTA-358555)

  • When the API endpoint /interact was called by an OAuth service client, admins received an invalid request error instead of an unauthorized client error. (API - OKTA-350666)

Release: 2021.02.0

Deployment date: February 3, 2021

Features and enhancements

  • Identity Engine now supports Active Directory (AD) delegated authentication. When delegated authentication is enabled, AD user data is used to authenticate users. Delegated authentication simplifies user management by making AD the single source of truth for user data and access credentials. See About delegated authentication with Active Directory.
  • The org-wide limit for app sign-on policies has increased. Admins can now configure unique sign-on policies for 5,000 different applications. See App sign-on policies.
  • Validation error messaging for OAuth Authorization and Interaction Code token exchanges is now more descriptive and corrects some faulty messaging.
  • Okta admins can now create app-based sign-on policies for the Okta Dashboard, Okta Admin Console, and Okta Browser Plugin.

    Previously, sign-on policies couldn't be configured for these first party applications. With this release, policy based on context such as user location, device, behavior, risk level, group membership, and more is included. This gives admins more flexibility and granular control over sign-on requirements for these first party apps. For example, different MFA requirements might apply to the Okta Admin Console for different groups of people. See Enable the new Okta End-User Experience.

  • During end user authentication, the Sign-In Widget failed if a RADIUS sign-on policy rule was present in the Okta sign-on policy. (OKTA-351575)
