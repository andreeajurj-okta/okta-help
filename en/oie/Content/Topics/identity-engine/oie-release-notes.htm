Identity engine release notes

Release notes for Identity engine are published here and include new feature releases, enhancements to existing features, bug fixes, and a history of past releases.

  • Latest release
  • 2021 releases
  • 2020 releases

Release: 2021.02.0

Deployment date: February 3, 2021

Features and enhancements

  • Identity engine now supports Active Directory (AD) delegated authentication. When delegated authentication is enabled, AD user data is used to authenticate users. Delegated authentication simplifies user management by making AD the single source of truth for user data and access credentials. See About delegated authentication with Active Directory.
  • The org-wide limit for app sign-on policies has increased. Admins can now configure unique sign-on policies for 5,000 different applications. See App sign-on policies.
  • Validation error messaging for OAuth Authorization and Interaction Code token exchanges is now more descriptive and corrects some faulty messaging.

Bug fixes

  • The Sign out link was not displayed when an Active Directory (AD) user who was disabled in AD and active in Okta set the password using the Forgot Password link. (OKTA-362135)
  • During end user authentication, the Sign-In Widget failed if a RADIUS sign-on policy rule was present in the Okta sign-on policy. (OKTA-351575)
  • When end users were activated or deactivated in Active Directory, Identity engine didn't retrieve the updated status. (OKTA-360032)
  • Some labels and buttons on the Sign-In Widget were not translated. (OKTA-362345)
  • In the app sign-on policy rules, the condition And User’s IP is didn’t include the options not in zone and not in any zone. (OKTA-344165)
  • The app sign-on policy rule displayed an incorrect value when the behavior expression language change was canceled. (OKTA-358047)
  • API calls from the Okta Applications Sign-on tab returned a 404 error. (OKTA-360189)

Release: 2021.01.2

Deployment date: January 27, 2021

Features and enhancements

  • The Interaction Code grant type is now available to allow OAuth applications to use Okta Identity Engine. See Enable interaction code grant.

Bug fixes

  • To prevent the implementation of unsupported functionality, the following unavailable features have been removed from the Admin Console:

    • Lightweight Directory Access Protocol (LDAP)

    • Integrated Windows Authentication (IWA) integration options

    • Just-In-Time (JIT) functionality for Active Directory (AD)

    • Agentless Desktop Single Sign-on (ADSSO)

      • (OKTA-361552, OKTA-361864, OKTA-362382, OKTA-362399, OKTA-362403, OKTA-362754)

  • After an org migrated to Okta Identity Engine, the self-service account unlock option was displayed but couldn’t be edited since it's not yet supported. (OKTA-353627)
  • When creating a Profile Enrollment policy, the error message that appeared if an admin specified more than one group didn't clearly state that no more than one group can be specified per policy. (OKTA-315860)
  • End users weren't challenged to provide Multifactor Authentication (MFA) when accessing a Bookmark app protected by an app sign-on policy that required MFA. (OKTA-354669)
  • End-user authentication from the Sign-In Widget wasn't successful when some features were enabled and an existing Lightweight Directory Access Protocol (LDAP) interface sign-on policy rule was enforced. (OKTA-359415)
  • After org migration to Okta Identity Engine, sign-on policy pages associated with RADIUS or MFA-only apps (Active Directory Federation Services, Remote Desktop Protocol and Electronic Prescribing for Controlled Substances) displayed inapplicable options and sign-on policies couldn’t be entered. (OKTA-357784)
  • When attempting to manage assignments for First Party Apps, vague error messages were returned. (OKTA-337718)
  • The app sign-on policy rule wasn’t evaluated when the custom expression case for behavior attribute and behavior name didn’t match. (OKTA-356003)

Release: 2021.01.1

Deployment date: January 13, 2021

  • During Okta Verify enrollment, some end users were prompted to set up Okta Verify as an authenticator even though the Okta Verify enrollment process was already in progress. (OKTA-343972)

Release: 2021.01.0

Deployment date: January 7, 2021

Features and enhancements

  • Admins can now enable or disable Phone and Email authenticators for single sign-on using the authenticator Edit option. See Authenticator configuration.

  • Admins can now configure behavior conditions in their app sign-on policies. See Add an app sign-on policy rule.

  • The Sign-In Widget displays a polling message if the end user attempts to sign in when the server is in Safe Mode or Read-Only Mode, or during rate limiting.
  • Under Security in the Admin Console navigation, Authentication has been renamed to Okta Sign-on Policy.

Bug fixes

  • When Identity engine was disabled, users couldn’t remove the Email Authentication option from Extra Verification settings using the Remove button. (OKTA-352229)
  • When a new org was created, its first admin was unable to change their one-time password from the end user Settings page if Identity engine was enabled before their first sign-in. (OKTA-355639)
  • Some remediation responses didn't have the correct values for the Accepts and Produces headers. (OKTA-350029)
  • When attempting to sign in, end users received an error if they selected Remind me later on the password expiration warning. (OKTA-355885)
  • The First Party Admin App incorrectly contained a settings button when listed in the App Catalog. (OKTA-341084)
  • Security methods for certain versions of Safari web authentication didn't allow for biometric authentication. (OKTA-343930)
  • Self-Service Registration with inline hooks failed for some orgs. (OKTA-352737)

Release: 2020.12.2

Deployment date: December 23, 2020

Release summary

  • Office 365 is now available in Okta Identity Engine. You can create new Office 365 app instances and migrate your existing Office 365 app instances. See Office 365 sign-on rules options.
  • After a user has successfully authenticated, the redirect text on the Sign-In Widget now specifies the app name and the user’s identifier.
  • For some users, when a password reset failed, an Internal Server Error was incorrectly returned. (OKTA-352465)
  • Instead of using the values that admins mapped during user and group assignment, provisioning-enabled OIN apps prompted end users to provide their own enum-typed attributes. (OKTA-349421)
  • Some text and icons weren’t aligned on the app sign-on policy rules page. (OKTA-350948)
  • App admin wasn't displayed in the Assignment tab for first party apps even when these apps could be assigned to app admins through API. (OKTA-341956)

Release: 2020.12.1

Deployment date: December 16, 2020

Release summary

  • Under Reauthentication frequency in the app sign-on policy configuration, a time value that was set to reauthenticate a user wasn't accurately preserved after changes were saved. (OKTA-350698)
  • End users who were assigned the first party Admin Console app received a 404 error after clicking the app in the Okta End-User Dashboard. (OKTA-344970)

  • The Admin Console app could be partially deleted from the user profile application list. (OKTA-348623)


Release: 2020.12.0

Deployment date: December 9, 2020

Features and enhancements

  • End-user reauthentication is enabled when a user edits their profile, password, or language setting from the Okta End User Dashboard. See End-user settings.
  • In app sign-on policies rules, the Conditions sections have been renamed If and Then.
  • Admins can now configure risk conditions in their app sign-on policies. See Add an app sign-on policy rule or Risk Scoring.
  • In the System Log:

    The user.session.start event indicates when a user's session begins after they sign in.

    The policy.evaluate_sign_on event include rules for Okta sign-on policy and app sign-on policy evaluation.

    The user.authentication.auth_via_mfa event shows if a user authenticates with MFA.

Bug fixes

  • The icon for Email Authenticator wasn’t displayed correctly in the Sign-In Widget. (OKTA-312021)
  • When a user submitted an incorrect answer to a security question in the Sign-In Widget, the error message was not displayed correctly. (OKTA-345397)
  • When a registered Okta Verify user attempted to update their email address on their device, the Confirm email change address link failed. (OKTA-346557)
  • When some user passwords expired, Sign-In Widget functionality was affected and didn’t display an error message. (OKTA-349016)
  • In Security > Authenticators > Setup, an error message appeared when admins tried to update User Verification settings in the Security Key or Biometric authenticator dialog box. (OKTA-348318)
  • Some federated users received an HTTP 500 internal server error when attempting to sign in to orgs. (OKTA-343930)
  • When admins viewed the application configuration for OAuth service clients, the Assignments tab appeared incorrectly. (OKTA-344880)
  • When a user was activated through the Okta Users API, the sendEmail parameter was ignored. (OKTA-345352)
  • In Profile Enrollment policy rules, admins could add users to groups that are not sourced by Okta. (OKTA-345832)
  • When an end user’s username didn’t match a routing rule, they received a permission error followed by a password error. (OKTA-349975)
  • During self-service password reset, users who were denied access by a sign-on policy couldn’t change to a valid username in the same flow. (OKTA-321603)
  • Signing in to SWA apps didn't work in some instances. (OKTA-323092)
  • Users who signed in and were prompted to verify with MFA didn’t have the option for security key requiring user verification if a possession factor was already verified. (OKTA-325963)
  • Users who accessed an app immediately after signing in were sometimes required to reenter their password even if the password reauthentication frequency in the app sign-on policy was set to two hours. (OKTA-346486)

Release: 2020.11.3

Deployment date: December 2, 2020

Release summary

  • When the Okta server is in safe mode, users who sign in to an app are not prompted to enroll in an authenticator if the app authentication requirements are met. See Safe mode . (OKTA-337488)
  • Email links for Self-Service Password Reset sometimes directed users incorrectly to the sign-in page. (OKTA-324362)
  • Users who were denied access by a sign-on policy couldn’t change to a valid username in the same flow. (OKTA-325589, OKTA-325728)

Release: 2020.11.2

Deployment date: November 18, 2020

Release summary

  • The Authenticators UI now includes a description of factor types, factor characteristics, and a link to admin documentation.

  • The Self-Service Password Reset feature failed for some users who attempted to reset their password. (OKTA-338494)
  • When a user entered an invalid one-time password (OTP) for email or phone authenticators, a poorly formatted error message was returned. (OKTA-345553)
  • App sign-on policies that were originally set to “Deny” couldn’t be saved as “Deny” after migration. (OKTA-330756)
  • On the Profile Enrollment page, the link to view more apps didn't work and the Add Apps to this Policy list didn't scroll. (OKTA-345082)

Release: 2020.11.1

Deployment date: November 11, 2020

Release summary

  • When registering AD/LDAP agents, users were sent to the end user dashboard before the registration flow was complete. (OKTA-318387)
  • When users attempted to sign in with just their name instead of full email address, in some instances the sign-in request failed. (OKTA-338834)

  • New users who created a profile without enrolling in a factor couldn't resume the registration process. (OKTA-317847)

  • New users who didn’t activate their account still received an email to enroll in one or more authenticators. (OKTA-329686)


Release: 2020.11.0

Deployment date: November 4, 2020

Features and enhancements

  • Okta sign-on policies now support custom expression language. When configuring an Okta sign-on policy rule, you can define a condition by entering a custom expression in the field The following custom expression is true. See Sign-on policies and rules.
  • The app sign-on policy is now only evaluated when a user accesses an app from a valid session. The Okta sign-on policy and app sign-on policy are both evaluated only when a session is not valid.
  • Users can now enroll in multiple WebAuthn security keys from the End User Settings page.

Bug fixes

  • The magic link authentication email didn’t specify that end users can also authenticate with a passcode if required. (OKTA-256452)
  • During authenticator enrollment, the Return to Authenticator List link appeared only if multiple authenticators were available. (OKTA-315132)
  • When a user clicked the Forgot password link and the request was denied, a misleading error was displayed. (OKTA-316798)
  • When users tried to access an app that required email authentication, the email didn’t state the app name. (OKTA-316989)
  • End users who clicked the Forgot password link on the sign-in page were not prompted to reset their password. (OKTA-336529)
  • When admins attempted to sign in to the Admin Console, the MFA OTP token expired after only two minutes. (OKTA-338832)

Release: 2020.10.2

Deployment date: October 21, 2020

Release summary

  • For migrated orgs, the Self-Service Account Unlock option was displayed under Password Policies when a user clicked a rule. (OKTA-323269)

  • Self-Service Password Reset with SMS OTP failed when initiated by a user before their org was migrated to Identity engine. (OKTA-324466)

  • When users self-registered with Okta from an app that didn’t require email verification, apps without email verification could still be accessed even if the saved Profile Enrollment Policy required email verification for those apps. (OKTA-331445)

  • In some cases, end users signing in to applications were prompted to enroll in MFA before being prompted to enter their password. (OKTA-324430)


Release: 2020.10.1

Deployment date: October 14, 2020

Release summary

  • The System Log now lists events for Self Service Password Reset. These events log when and how an end user attempted to reset their password.

  • Users who clicked an expired email link were unable to navigate back to the sign-in page. (OKTA-318058)

  • Users who attempted to set passwords that didn’t meet password requirements were shown errors that contained brackets. (OKTA-317654)

  • Users couldn’t sign in if their accounts were enrolled in multiple WebAuthn authenticators. (OKTA-334594)
  • The MFA enrollment prompt was shown prior to the password prompt when some users attempted to sign in to their accounts. (OKTA-323520)

Release: 2020.10.0

Deployment date: October 7, 2020

Release summary

  • When enrolling an authenticator, the Setup authenticator page incorrectly displayed the error message “Sign in using something else” for the use of a different authenticator. (OKTA-320266)

  • Users received a complex error message that listed all constraints when minimum password requirements weren't met. (OKTA-326110)

  • The Email Challenge template didn’t include the magic link. (OKTA-328538)


Release: 2020.09.4

Deployment date: September 30, 2020

Release summary

  • Entering an invalid phone number when enrolling a phone authenticator generated an error.

  • The Remember Me field now appears after the Password field on the Okta Sign-in Widget.

  • The Forgot password? link now appears on the Sign In page whenever a user is prompted to enter a password. If the user tries to reset their password and the attempt fails, the failure is now logged in the System Log.


Release: 2020.09.3

Deployment date: September 23, 2020

Release summary

  • The Send lockout email to user functionality for self-service account recovery in Password Policy is not available in Identity engine.
  • Conditions to exclude users and groups were not preserved when app sign-on policies were migrated from Okta Classic to equivalent policies in Identity engine.
