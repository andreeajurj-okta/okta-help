Device Trust on Okta Identity Engine known issues

 

This page is updated frequently.

The following are the known issues with Device Trust on Okta Identity Engine:

Platforms

Issue

All platforms

 

  		Before deleting Okta Verify from a device, end users should first remove accounts from the app.

Deactivated users get an error when deleting an account in Okta Verify

If a deactivated user tries to delete an account in Okta Verify, they get an error Resource not found. To resolve, end users must contact their IT administrator.

Suspending then reactivating a user

If you suspend and then reactivate a user, they can’t enroll Okta Verify on a different device. To resolve, end users must contact their IT administrator.

Not possible to access an app without enabling biometrics in some circumstances

In some cases, end users who haven’t enabled User Verification (biometrics, Windows Hello) for their Okta Verify account can’t access Okta-protected apps.

Scenario

If a user does all of the following:

  1. Installs Okta Verify on their device;
  2. Adds an account to Okta Verify but does not enable Extra Verification (Windows Hello) for the account at that time or later;
  3. Tries to access an Okta-protected app from:
    • The Okta End-User Dashboard and they didn’t use Okta Verify to sign in to the dashboard (IdP-init).

      • - or -

    • The same browser, and in the same browser session, from which they added an account to Okta Verify (SP-init).
  1. And the app is gated by an app Sign-On policy in which all the rules require that:
    • The user provide a possession factor
    • The re-authentication frequency is set to either Every sign-in attempt or Re authenticate after N minutes/hours/days
  1. Enters their email address instead of clicking Sign in using Okta Verify at the following screen:

One of the following known issues occurs:

  • Instead of accessing the app as expected, the user is denied access and gets the error message Unable to fully authenticate into appinsstance Id <>.

    • -or-

  • If the user is shown a list of authenticators, Use Okta Verify on this device isn’t included because the user hasn’t enabled biometrics for their account. Without the Use Okta Verify on this device option in this case, the user can’t access the app.
The user sees this: Instead of this:

Workaround

Any of these admin or end-user steps can prevent or correct the issue:

Admins

Set the re-authentication frequency to Once per device for all rules in the app’s Sign On policy.

- or -

End users

Do any of the following:

  • Click Sign in using Okta Verify when accessing the app from the Okta End-User Dashboard.
  • Enable Extra Verification for their Okta Verify account (see Enable Extra Verification for an existing account).
  • Access the app directly (not from the Okta End User Dashboard) but make sure to use a different browser – or a different browser session – than they used when they added an account to Okta Verify.

Polling loop if user declines to provide required biometrics during app access

When accessing an app that requires user verification (biometrics or PIN) to access, if the user clicks or taps Cancel when prompted for user verification, Okta Verify continues to prompt the user for user verification instead of redirecting to the org's home page. To remedy, the user can try the following: 

  • Close and then reopen Okta Verify, and try to access the app again.
  • Go to Okta Verify settings and turn User Verification off then on again.
  • Remove and then re-add the account in Okta Verify.

Configure Device Management before creating app sign-on policies that require devices to be managed

When accessing an app protected by a policy that requires devices to be managed, users get an internal server error If Device Management is not already configured in Security > Device Management. Before you create an app sign-on policy that requires devices to be managed, make sure to configure Device Management.

Error if user enters the wrong email when setting Okta Verify through email link

When setting up Okta Verify, if an end user chooses the Set up Okta Verify via email link option instead of scanning a barcode, they must enter either the primary or secondary email address in their Okta profile. Entering any other email address generates an error.

List of factors includes options inappropriate for the situation

If a user doesn't enable user verification (biometrics) when enrolling in Okta Verify and then tries to enable it after they’ve enrolled, they need to provide a factor to do so. The list of factor options shown to the user in this case should include only Password and Security Question factors but the actual list includes Push and TOTP factors which aren’t appropriate in this case. Users who still want to use Push and TOTP factors in this case can try:

  • Push: Accept the push notification instead of the in-app notification during Okta Verify enrollment.
  • TOTP: Go to the Okta Verify accounts page and tap the six-digit code to copy it. Return to the Account Details page to enable This device as a sign-in method. Enter the six-digit code when prompted.

List of factors includes an option to authenticate with Okta Verify when setting up Okta Verify

When enrolling in Okta Verify, end users may be prompted to set up Okta Verify as an authenticator even though Okta Verify enrollment process is already in progress, which may cause confusion.

MDM remediation message is not shown in some cases

If you've configured an app sign-on policy that requires devices to be managed but have not configured Device Management (Security > Device Management), users on unmanaged devices are shown the message You do not have permissions to perform requested action instead of a message guiding them to enroll in your chosen MDM vendor.

User's Devices tab limited to displaying only 20 devices

The Devices tab in a user's profile can display only up to 20 devices even if the actual number of devices linked to a user exceeds 20.

Removing Okta Verify from a device must be done in the correct order

Uninstalling Okta Verify from a device in the wrong order breaks the connection between the user’s local Okta Verify registration and the corresponding registration maintained on the Okta Server. If the connection is broken, the Okta server is unaware of the change and continues to consider the registration intact, which prevents the user from resetting their enrollment or creating a new enrollment on the original device or another device without assistance from their IT admin.

To uninstall Okta Verify from a device, users should perform these steps in the following order:

  1. From a browser, access their Okta End User Dashboard, go to Settings, scroll to Extra Verification, and then tap or click to Remove Okta Verify. This removes the registration from the Okta server.
  2. On the device on which Okta Verify is installed:
    1. Go to Okta Verify settings and delete any Okta Verify accounts from the app.
    2. Locate the Okta Verify icon, long press (mobile) or right-click the icon (desktop), then click or tap Uninstall.

Okta Verify enrollments still active after admin uses Revoke access option

Users can still use Okta Verify to access protected resources even after their admin removes the relationship between the user and the device by revoking access through the Okta Admin Console. If you intend to remove a user's Okta Verify authenticator enrollment on a device, go to the user's profile page, click More Actions, click Reset Authenticators, select Okta Verify, and then click Reset Selected Authenticators.

Adding new biometrics to user device breaks user verification enrollments in Okta Verify

If a user who is enrolled in Okta Verify with user verification (biometrics) changes their device biometrics, their existing user verification enrollment breaks. The user must disable user verification in Okta Verify Account Details by signing in to their account and then re-enabling user verification to get back to a good state. If the user isn't enrolled in any other authentication method, they may need to contact their IT admin for assistance.
Windows

 

 

In orgs configured to require 2FA, users can remove their Okta Verify enrollment with 1FA

Given an org configured to require users to provide 2 factors to remove an OV enrollment:

When a user tries to remove their Okta Verify enrollment through the settings page in their Okta end user dashboard, they’re prompted to provide:

(a) A password and a biometric factor if the user enabled Extra Verification (biometrics) for their Okta Verify account.

-or-

(b) Only a password if the user didn’t enable Extra Verification (biometrics) for their Okta Verify account.

Because the org requires 2FA, (b) may be unexpected.

Custom URL doesn't appear when adding an account in Okta Verify

If a user specifies their org's custom URL when they add an account in Okta Verify, the account appears with the org's Okta URL instead of the org's custom URL.

Deactivated users can't delete Okta Verify accounts

If a deactivated user tries to delete an account in Okta Verify, they get an error Resource not found. The user must contact their IT administrator.

macOS and Windows

End users are sometimes prompted to install Okta Verify when it's already installed

End users who have already Okta Verify installed on their desktop device may be prompted in error to install Okta Verify again when they try to access certain apps. This can happen if a user hasn’t enabled user verification (biometrics) in Okta Verify and the app they’re trying to access requires user verification. To avoid getting stuck in a loop, the end user must either:

  • Enable user verification in Okta Verify by clicking the three vertical dots next to the account, and then click Enable Touch ID (mac) or Windows Hello (Windows), then try to access the app again.

    • – or –

  • Install Okta Verify on a different device and enroll in Push and TOTP, then use this device to access the app.

Enrolling Okta Verify on multiple desktop devices constraints

  • To enroll in Okta Verify on multiple desktop devices, users must be enrolled already with another factor which can be used to sign in to the additional desktop, such as Yubikey, SMS, or Okta Verify with Push.
  • If the user's first Okta Verify enrollment is on a desktop device, their next Okta Verify enrollment must be on a mobile device. For end user enrollment, see Get started with Okta Verify.

If a user tries to enroll Okta Verify on a second desktop device without following the above constraints, they’re prompted repeatedly to enroll using Okta Verify, which is not possible in this circumstance.
iOS

Open Verify doesn't open in some cases

Instead of opening Okta Verify as expected, trying to open Okta Verify from a Chrome incognito tab on an iOS 13.x-device already enrolled in Okta Verify redirects users to the Download Okta Verify page.

Some Okta Verify accounts backed up on a device appear when restored to another device

Given

  • A user in an Okta Identity Engine org with Okta Verify installed on their iOS device.
  • The user adds two Okta Verify accounts in the app: one for an Okta Classic org and another for an Okta Identity Engine org.
  • The user backs up their device to iCloud or a macOS device.
  • The user restores this backup to a second iOS device.

Issue

Although neither Okta Verify account should appear on the second device, the account from the Okta Identity Engine org does appear but none of it's configured authentication methods work.

Workaround

On the second device, the user should delete accounts from Okta Verify and then add them again. See Manage Okta Verify accounts on iOS devices.

Authentication methods appear following backup-restore

If a user sets up Okta Verify on their device, backs up the device to iCloud or a macOS device, and then restores the backup to their device, no authentication methods should appear configured in Okta Verify Account Details. In this case, all methods do appear as configured but only Authentication Code works. To resolve the problem, users should remove the Okta Verify accounts and add them again. To remove an account, see Delete an account.

Users are prompted to set up Okta Verify as an authenticator erroneously when removing an Okta Verify account

Affected users should ignore the Set up required prompt by clicking Cancel on the setup screen. Their Okta Verify account is removed as expected.

Okta Verify doesn't open as expected in Chrome incognito

Trying to open Okta Verify from a Chrome incognito tab on an iOS device already enrolled in Okta Verify redirects users to the Download Okta Verify page instead opening Okta Verify, as expected.
Android

Google Pixel 4 XL devices prompt for fingerprint scan which isn't supported by the device

On Pixel 4 XL devices, users who have not enabled biometrics on their device are prompted by the device operating system to scan their fingerprint to complete Okta Verify flows that require user verification (biometrics). Pixel 4 XL devices support face ID, but not fingerprint scan.

Multiple Okta Verify accounts not supported in the same Okta Identity Engine org

  • Trying to add a second Okta Verify account in an Okta Identity Engine org – This is unsupported. If a user tries to add a second account in the same org, an error appears.
  • Accounts disappear if users have multiple Okta Verify accounts before their org was upgraded to Okta Identity Engine – If you upgraded from Okta Classic to Okta Identity Engine, users who had multiple Okta Verify accounts in the Okta Classic org will see at least one of those accounts disappear following the upgrade.
Android and iOS

In some cases, the Okta Verify configuration in a user's Okta End User Dashboard shows two Okta Verify enrollments instead of one.

The context of this issue is an Okta Classic org without Push enabled upgraded to an Okta Identity Engine org in which Push+TOTP are enabled by default.

Given

  1. An Okta Classic org on which Okta Verify with Push is not enabled
  2. A user is enrolled in Okta Verify
  3. The org is upgraded from Okta Classic to Okta Identity Engine
  4. The user sets up their device as an authenticator through Ways to sign in Okta Verify Account Details.

Issue

In the Settings > Extra Verification section of the user's Okta End User Dashboard, the user's Okta Verify configuration includes two Okta Verify enrollments instead of one, which may cause confusion.

Workaround

Through Settings > Extra Verification section of the user's Okta End User Dashboard, the user should remove the enrollment called “Okta Verify.”

In some cases, while setting up an enrolled device as an authenticator through Okta Verify, it's necessary to obtain a temporary passcode from the same Okta Verify instance to complete the setup

Users may have difficulty using Okta Verify on their mobile devices to set up that same device as an authenticator for passwordless authentication (that is, setting up the This device option in Ways to sign in in Okta Verify Account Details). This is because, while setting up This device in Okta Verify, the user is prompted to verify their identity by entering a temporary code generated by Okta Verify which would involve leaving the set up flow to get the temporary 6-digit code and then returning to the set up flow to enter the code before the code expires (approx. 30 seconds). As a best practice, end users in this situation should do the following:

  1. Before enabling This device as a sign-in method, users should check whether other sign-in methods are enabled. If only Authentication code is turned on, users should:
    1. Go to the Okta Verify accounts page and tap the six-digit code to copy it.
    2. Immediately return to the Account Details page to enable This device as a sign-in method.
    3. Enter the six-digit code when prompted.
Currently, Okta Mobile is not supported in Okta Identity Engine orgs.
