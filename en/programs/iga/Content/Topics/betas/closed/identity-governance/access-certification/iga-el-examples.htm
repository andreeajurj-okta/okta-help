Examples of Okta Expression Language

Okta Expression Language (EL) allows you to reference, transform, and combine user attributes and group information. See <Okta EL dev docs>. While creating or modifying an access certification campaign, you can use Okta EL expressions to define:

  • Users in order to restrict your campaign to a subset of users.

  • Reviewers to customize reviewers for each user.

Tips

  • Review the Okta Expression Language documentation to familiarize yourself with the syntax.

  • Use the ternary operator [Condition] ? [Value if TRUE] : [Value if FALSE] to build conditions for defining the dynamic reviewers.

  • Ensure that your expression evaluates to a boolean when defining users:

    • True to include the users.

    • False to exclude them from the campaign.

  • When defining reviewers:

    • Ensure that your expression evaluates to either the user ID or the username of a single Okta User. If the expression does not, the Fallback reviewer defined in the campaign builder will be assigned to review all items for that user.

    • It’s helpful to think of reviewer logic into IF/THEN terms for each user when building your expressions. For example, for user A, if condition P is true, then assign reviewer B.

  • Start with simple expressions and gradually add in conditions to make sure your expression works as expected.

  • Test your expression using the Preview functionality on the Users and Reviewers pane.

  • Use any value stored on a user’s profile and group to restrict the scope of a campaign.

  • Use either the Group ID or Group Name to reference a group in your expression. We recommend that you use Group ID as it doesn’t change.

  • Use

    • && to denote the And operator.

    • || to denote the Or operator.

    • ! to denote the Not operator.

Define user scope

Use Okta Expression Language to limit the scope of the access certification campaign to a set of users based on their profile attributes and group membership. Your custom expression must evaluate to true to include the users or false to exclude them from the campaign.

Ensure that your expression evaluates to a boolean: True to include the users or False to exclude them from the campaign.

Define dynamic reviewers

Use Okta Expression Language to restrict the set of reviewers assigned to review the users’ access. You can specify reviewers based on their profile attributes and group memberships. Your custom expression must evaluate to the Okta User ID of the user who should be assigned as the reviewer.

Tips

Use [Condition] ? [Value if TRUE] : [Value if FALSE] as a ternary operator to build conditions.

Scenario one

As an admin, you want to set up a campaign to review multiple apps for your sales team where a user’s manager should review the user’s access to all apps. In this case, you can use the following EL expression:

user.ManagerId

Scenario two

As a Salesforce admin, you are running a campaign for Salesforce and want to specify a different reviewer for users in each department.

  • For the Finance department, the user’s manager should review the user’s access.
  • For the Sales team, the Head of Sales Ops should review the team member’s access.
  • For interns, the Intern Coordinator should review intern’s access.
  • For everyone else, the Salesforce admin should review users’ access.

You have the following setup for your org:

  • Department is a user attribute:
    • All sales departments must have the word ‘sales’ in their name. For example, Enterprise Sales, Commercial Sales, and Sales Operations, and so on.
    • Finance is listed as the department for all Finance team members.
  • Manager and Manager ID is stored in the Okta user attribute.
  • Interns are set up as a group in Okta with the name Interns (Group Id: 007haGt87GTWghj67YUAS).
  • The Head of Sales Ops is Jane Smith (User ID: 00u12u3wri8DS5JTD1d7).
  • The Intern Coordinator is Michael Thomas (User ID: 00u12u3wri8hua87GhtK).

In this case, you can use the following EL expression:

String.stringContains(user.department, "Sales") ? "00u12u3wri8DS5JTD1d7" : (user.department == "FINANCE" ? user.managerId : ( users.isMemberOfGroup("007haGt87GTWghj67YUAS") ? "00u12u3wri8hua87GhtK" : null))

Also, you should set the Fallback Reviewer to yourself as the Salesforce admin. All users without a specified reviewer will be assigned this Fallback Reviewer. In this case, it is anyone outside of the Sales or Finance Departments and who isn’t an Intern.

Scenario three

As an app admin, you want to run a campaign with blended reviewer requirements. For most users, the user’s manager should review the user’s access. However, for users who report to someone in the C-suite, the app admin should review their access.

You have the following setup for your org:

  • Manager and Manager ID is stored in the Okta user attribute.

  • Executive staff is a user group (Group ID: 007haGtKYZTWghj67STSK) that contains all C-suite Members.

In this case, you can use the following EL expression:

(user.manager.isMemberOfGroup(“007haGtKYZTWghj67STSK”)) ? null : user.managerId

Also, you should set the Fallback Reviewer to yourself as the app admin.

This EL evaluates as:

If a user’s manager is a member of the Executive Staff, don’t return a reviewer. Then, this will default to the Fallback Reviewer. Otherwise, assign the user’s manager.
