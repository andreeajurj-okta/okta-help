Customization

You can the following:

User Account

If your users manage their accounts outside of Okta, you can replace the Okta user account forms (personal information and password) with a custom message and redirection link to the external application managing these credentials. After you enable this, users can access these links from the Account tab on their My Applications or "home" page. They can also access them by clicking the Forgot password? link on the Okta Sign-In page.

You can also customize your users' Account page based on which system manages their identity (for example, Active Directory, SunOne, etc.). A user can only have one identity master at one time.

  1. Click Edit in the User Account section. If different users are mastered by different systems, you see the User Identity Master drop-down list, which enables you to configure different settings based on which systems manage your users.
  2. If applicable, select Pesonal information is managed by Okta. If personal information is managed outside of Okta by another application, do the following:
    1. Select Personal information is managed by a different application.
    2. Enter a custom message.
    3. Enter a link label.
    4. In the Custom link URL field, enter the redirect URL of the website you want users to visit to change their password.

      Important:

      • Your URL must specify https (http will not work).
      • Due to a lack of broad support across browsers, do not use X-Frame-Options to whitelist your Okta domain.
  3. If applicable, select Password is managed by Okta. If passwords are managed outside of Okta by another application, do the following:
    1. Select Password is managed by a different application.
    2. Enter a custom message.
    3. Enter a link label.
    4. In the Custom link URL field, enter the redirect URL of the website you want users to visit to change their password.

      Important:

      • Your URL must specify https (http will not work).
      • Due to a lack of broad support across browsers, do not use X-Frame-Options to whitelist your Okta domain.
  4. Click Save.
Optional User Account Fields

This section allows for optional fields to appear for your end users that allow for account creation notification and a security image.

Configuring Your Secondary Email Option

As an admin, you can set a secondary email option for your end users. If you prefer that password reset and account creation notifications are sent only to your end user's' primary email address, the Secondary Email option can be disabled.

To disable the secondary email option, click Edit in the Optional User Account Fields section, and then select Disable on the Secondary Email drop-down list, and click Save when done.

 

Customizing Your Security Image

You can accompany an Okta login by an optional security image feature to give Okta end users additional protection from phishing attacks.

Enabling a Security Image

To customize your security image, click Edit in the Optional User Account Fields section.

Select Enabled or Disabled for the Security Image option, and then click Save.

Image Selection

When a user first logs into an Okta org with a security image enabled, the user is required to select an image. At this time the browser receives a cookie containing the selected security image. The cookie is signed by the site certificate to prevent an attack site from accessing and loading the security image, since an attack site does not have the correct certificate.

Image Display

When a user enters a username in the Sign In screen, and tabs or clicks to move to the Password field, a security image displays. When users recognize their selected security image, they are reassured that they are logging into the same site they previously accessed. If they do not see the image, it might mean that they're looking at a spoofed attack page.

Limitations

Since the browser must first create the cookie before displaying the image, the security image is not displayed in the following circumstances.

  • First login
  • First login from a new browser or device
  • First login after clearing cookies

To enhance security, Okta recommends using Multifactor Authentication (MFA) as a further layer of Security.

Okta User Communication

You can optionally prevent Okta from communicating with your end users by changing your Okta User Communication settings by clicking Edit in this section, selecting Opt Out of Okta User Communication, and finally clicking Save.

Deprovisioning Workflow

Some apps must be manually deprovisioned. The Deprovisioning Task Manager helps admins keep track of these tasks, which are indicated under Status on the Administrator Dashboard. You can opt to disable the deprovisioning workflow. This means that you will not be alerted to tasks or receive emails when you deactivate or unassign an application from a user.

To disable your deprovisioning workflow:

Click Edit in the Deprovisioning Workflow section, and deselect Enable Deprovisioning Workflow. When done, click Save.

Browser Plugin

You can configure browser plugin settings to manage plugin installations and upgrades, as well as some browser behaviors. This option is useful in locked-down environments where end users can't install or manage the Okta Plugin on their computers.

To configure browser plugin settings:

Click Edit in the Browser Plugin section and set the following items:

  • IT manages Okta Plugin: Select Yes to enable; No to disable. If enabled, the messages prompting users to install or upgrade the plugin are hidden.
  • Enable Okta toolbar for group: Specify the group that can use the toolbar to access their app outside of Okta.
  • Block browsers from saving passwords for Okta apps: Select Yes to enable; No to disable. Enabling this setting prevents save password banners from displaying when end users open apps from their Okta dashboards.
  • Block browsers from saving Okta password: Select Yes to enable; No to disable. Enabling this setting prevents save password banners from displaying when end users log in to their Okta home page.
  • Warn when visiting new organizations: Select Yes to enable; No to disable. Enabling this setting displays a warning message in the browser when an Okta user attempts to log in to an unknown org. This security feature prompts end users to use caution before they enter their login credentials for an org that is not their primary org.

Click Save.

Display Language

You can specify a default language preference for the end user experience of your entire org. Additionally, your end users can choose a language preference for their own experience. The end user's preference overrides the org default language.

To change the default language for your entire org:

  1. Click Edit in the Display Language section. The Language preference dropdown menu becomes active.
  2. Select the language for end users in your org.
  3. Click Save.

In your end users' dashboards, all the user interface labels and buttons are now in your preferred language.

Note: The following are not localized:

  • Admin pages (with the exception of the Email Templates)
  • Customized labels in the Sign-In Page, if they are in a language other than what is specified in Display Language

For more details, including instructions on how end users can configure their own language preference, see Setting Language Preferences.

Sign-In Page

The Sign-In page is the first element of Okta your end-users see. You can change the headings and customize links that appear on this initial page.

Note: For the sign-in page to display correctly, your browser must be at least 750px in height.

In the Sign-In Page section, click Edit and configure or change the headings and customize the links that appear on your sign-in page. This includes the sign-in heading username fields (including inline label), password (including inline label), forgotten password and help links, and two custom link fields for you to add your own links.

When finished, click Save.

Tip: If the language of your custom labels differs from the language selected in Display Language, your Sign-In Page will be in two languages. To avoid this, make sure your custom labels are in the same language as the specified Display Language. To correct dual languages on the Sign-In Page quickly, click Reset values to replace all customized labels with default labels in the specified Display Language.

Sign-Out Page

The sign-out page is the last element of Okta your end-users see. You can change the URL for this page.

  1. In the Sign-Out Page section, click Edit, and select Use a custom sign out page.
  2. Add the URL for your custom page, and click Save.
Application Access Error Page

If end users try to access an app to which they are not assigned, you can configure Okta to redirect them to the default Okta URL or to a custom URL that you provide. Unassigned users are more likely to try to access apps if you embed app links in your portal or other sites outside of Okta.

Note: To specify a custom error page for an individual app, see Using the Okta Applications Page.

To specify a custom error page for all apps, do the following:

  1. Click Edit.
  2. In the Application Access Error Page section, select one of the following options:
    • Use the default Okta error page

    • Use a custom error page

      For custom error pages, enter a URL to redirect unassigned users to a custom error page. This setting applies to your entire org. Optionally, you can configure a different URL on a per-app basis. An app-specific URL overrides the org-level URL that you specify here. For details, see Using the Okta Applications Page.

  3. Click Save.
