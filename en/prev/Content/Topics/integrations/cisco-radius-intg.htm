Part 3 – Configure Palo Alto Global Connect to use the Okta Palo Alto Radius Agent

In Part 3 you define a Radius Server Profile, define an Authentication Profile for Okta Palo Alto RADIUS Agent, apply the Okta RADIUS Authentication Profileto a Gateway, and configure the GlobalProtect Portal to use the Okta RADIUS Authentication Profile. Complete these using the Cisco ASA Admin Console.

Step 1 – Create AAA Server Group

  1. Log in to the Cisco ASDM console for the VPN appliance
  2. Navigate to Configuration > Remote Access VPN > AAA/Local users > AAA server groups, as shown below.

  3. Click Add to create a new group. The Add AAA Server Group screen opens, as shown below.

  4. Leave the default settings except for the following

    • AAA Server Group – specify a name to identify the group for the MFA server

    • Protocol – select RADIUS if necessary

  5. Click OK to return to the Cisco ASDM console, shown in step 2, above.

  6. In the AAA Server Groups list, select the server group you just created.

  7. In the Servers in the Selected Group pane, click Add. The screen shown below opens.

  8. Leave the default settings except for the following
    • Interface Name – select the interface that will handle communication with the MFA Server
    • Server Name or IP Address – specify the name or the IP address of the Okta RADIUS Agent
    • Timeout (seconds) – 60 seconds
    • Server Authentication port – enter the port number you configured above
    • Server Accounting Port – (not relevant, accounting not supported)
    • Retry Interval – leave default at 60 seconds
    • Server Secret Key – provided secret defined above
    • Common Password – re-enter the passphrase
  9. When done, click OK.

  10. Click APPLY to save the configuration.

  11. After saving you can perform tests by clicking the Test button.

Part 5 – Configure the GlobalProtect Portal for the Okta RADIUS Authentication Profile
  1. Navigate to Network > GlobalProtect > Portals and open your configured GlobalProtect Portal, as shown below.

  2. Select the Authentication tab to define Client Authentication Settings.
  3. Click Add to update Client Authentication to the Okta RADIUS Authentication Profile you just configured.

Part 7 – Configure Optional Settings

There are two optional settings you can configure, Client IP Reporting and Groups response.

Configure Client IP Reporting

The Cisco ASA Gateway does not report the Client IP address by default. For Devices running PAN-OS 7 and greater, you can configure the device to send this information with the following command from the devices administrative shell.

set authentication radius-vsa-on client-source-ip

The Cisco ASA NGFW does not Send the Client IP using the standard Attribute Value Pairs (AVP) like 31 (Calling-Station-Id); instead it sends the data using a Vendor Specific Attribute (VSA). To configure Okta to be able to parse, report on, and eventually enforce policy based off of the source client IP Address, configure the Okta Palo Alto Radius App in the Okta Admin Console, as shown below.

Enter the following settings in Advanced RADIUS Settings.

  • Client IP: Check Report client IP
  • RADIUS End User IP Attributes: 26 Vendor-Specific, 7

Configure Groups Response

The Palo Alto Network Gateway does not receive groups using the standard Attribute Value Pairs (AVP) of 11 (Filter-Id) and 25 (Class). Instead it relies on a Vendor Specific Attributes that Okta does not support.

For more information onhow to leverage functionality from Vendor Specific Attributes not supported nativly in Okta, please contact your Okta Customer Support Manager or Account Representative.

Troubleshooting

There are two troubleshooting procedures, Review the Authentication Logs and Test the Authentication Profile.

Review the Authentiction Logs

  1. Open a web browser and navigate to the GlobalProtect Admin Page, or open the Palo Alto Networks Panorama.
  2. Navigate to Monitor > Authentication and enter subtype eq Radius in the search filter, as shown below.

  3. Open the Palo Alto Networks Administrative Shell and run the following commands:

    debug authentication on dump

    tail follow yes lines 20 mp-log authd.log

  4. Click enter to run the test. Review the logs for detailed output.

Test the Authentication Profile

  1. Open the Palo Alto Networks Administrative Shell and run the following commands, as shown below

    test authentication authentication-profile [profile_name] username [username] password

    Substitute [profile_name] and [username] with your values for the profile name and the username.

    Note: The command line test of the Authentication Profile does not support a second factor challenge; the fact that you received a second factor challenge is validation that the configuration is working.
