Enforce Okta Device Trust for Chrome and Safari on Jamf and AirWatch-managed macOS devices

 

Okta Device Trust for Chrome and Safari on Jamf and AirWatch-managed macOS devices allows you to prevent unmanaged macOS devices from accessing enterprise services through Chrome and Safari browsers. It works with SAMLAn acronym for Security Assertion Markup Language, SAML is an XML-based standard for exchanging authentication and authorization data between an identity provider (IdP) and a service provider (SP). The SAML standard addresses issues unique to the single sign-on (SSO) solution, and defines three roles: the end user, the IDP, and the SP. and WS-Fed apps that can access the certificate store on the device when performing the federated authentication flow to Okta.


Key benefits
  • Ensures that only Jamf- or AirWatch-managed macOS devices with an Okta Device Registration Task can access SAML and WS-Fed cloud apps from Chrome and Safari browsers
  • Provides a frictionless end user experience by utilizing the Okta Certificate Authority
Requirements
  • Works on devices running Okta-supported versions of macOS:
  • Works with jamf PRO or VMWare AirWatch MDM solutions
  • Works with any SAML or WS-Fed cloud appAn abbreviation of application. Essentially, it is a web-based site used to perform any number of specific tasks, and requires authentication from end users by signing in. accessed from Chrome and Safari browsers

Part Ⓒ — Add the modified Okta Device Registration Task to your MDM app and distribute it to macOS devices

jamf PRO

  1. In jamf PRO click the gear icon to go to All Settings.
  2. Go to Computer Management > Scripts.
  3. Create a new script and paste into it the Okta Device Registration Task you downloaded from Okta in Part Ⓐ and modified in Part Ⓑ.
  4. Save the script.
  5. Create a policy to deploy the script to one or more Target Computers and/or Target Users:
    1. Go to Computers > Policies.
    2. Click General.
    3. Give the policy a name.
    4.  Select the following Trigger events that will initiate the policy.
      • Login – the script runs when an end user logs in to their macOS device.
      • Enrollment Complete – the script runs immediately after the devices complete the enrollment process.
      • Recurring Check-in – the script runs according to the recurring check-in frequency configured in jamf PRO.
    5. Set the Execution Frequency to Once per computer per user.
    6. Click Scripts and add the script you created earlier to this policy.
    7. Click ScopeA scope is an indication by the client that it wants to access some resource..
    8. In Target Computers and/or Target Users, specify the computers and users to whom the policy will apply.
    9. Save the policy.

AirWatch

  1. Go to Devices > Staging & Provisioning > Components > Files/Actions.
  2. Click Add Files/Actions.
  3. On the next page, click the macOS tile.
  4. In the General tab, enter, at minimum, a name for the manifest.
  5. In the Files tab, click Add Files.
  6. Click Choose Files to browse to the Okta Device Registration Task you downloaded from Okta.
  7. Click Save.
  8. Upload the script to AirWatch: In the Download Path field, enter the location where you want to download the script on end user macOS devices. Okta recommends that you specify tmp/MacOktaDeviceRegistrationTaskSetup.x.x.x.sh or any location from which the script can be removed automatically after the device is enrolled. Tip: Copy the path to your clipboard so that you can enter it in again in Step 12.
  9. If necessary, enter the version number for the manifest.

    AirWatch automatically numbers your first manifest version 1.

  10. In the Manifest tab, click Add Action.
  11. In Action(s) To Perform, select Execute Script.
  12. In Script File Path and Name, enter the same path you specified earlier in the Download Path field.

    Important: Do not select the option Execute as Root.

  13. Click Save.
  14. Click Add Action again.
  15. In Action(s) To Perform, select Run.
  16. In Command Line and Arguments to run, enter rm -rf/tmp/MacOktaDeviceRegistrationTaskSetup.x.x.x.sh. This ensures that the registration task is removed after it is executed.
  17. Click Save.
  18. Go to Devices > Staging & Provisioning > Product List View.
  19. Click Add Product.
  20. Click the macOS tile.
  21. In the General tab enter the name of the manifest you created previously.
  22. In Assigned GroupsGroups allow you to organize your end users and the apps they can access. Assigning apps to large sets of end users is made easier with groups., select the users or group(s) to whom you want to assign the manifest.
  23. Click the Manifest tab and then click Add.
  24. In Action(s) To Peform, select Install Files/Actions.
  25. In Files/Actions, choose the manifest you created previously.
  26. Click Save.
  27. Click Activate.
  28. If you later need to update the script on target devices, you can go to Staging & Provisioning > Product List View > More Actions and click Reprocess or Force Reprocess as appropriate for your use case. For details, see AirWatch documentation.

Part Ⓓ — Configure app Sign On policy rules in Okta

Overview

By default, all ClientEssentially, a client is anything that talks to the Okta service. Within the traditional client-server model, Okta is the server. The client might be an agent, an Okta mobile app, or a browser plugin. options in the App Sign On Rule dialog box are pre-selected. Notice that the Trusted and Not trusted options in the Device Trust section are not available unless you deselect all options in the Client section except the following, as applicable:

  • Web browser or Modern Auth client
  • iOS and/or Windows and/or macOS, as applicable

To configure more granular access to the app, selectively apply conditions as you create one or more prioritized rules based on:

  • Who users are and/or the groups to which they belong
  • Whether they are on or off network or within a defined network zone
Whitelist and blacklist approaches to creating Sign On policy rules

Typical approaches to creating and prioritizing rules often employ whitelist and blacklist concepts:

  • In a whitelist approach, you might do the following:
    1. Create one or more permissive rules to support the scenarios that will allow access to the app, then assign those rules the highest priority.
    2. Create a Deny catch-all rule that will apply to users who don't match the permissive scenarios you created in Step 1. Assign the Deny catch-all rule the lowest priority, just above Okta's Default Rule. The Default Rule is applied to users who don't match any of the more restrictive rules that have higher priority.
  • In a blacklist approach, you create one or more restrictive rules designed to deny app access to — or increase the authentication requirements for (MFA, Device Trust) — everyone except certain users/groups connected to a specified network or network range from specified client(s) and platform(s). Users able to pass through the restrictive rule(s) are then subject to the Default policy, which is designed to allow access to everyone.

Part Ⓔ — (Optional) View logs

To watch and monitor Okta-related log messages during certificate enrollment, be aware that the method differs depending on the operating system running on the device.

If running version 10.11.6 (OS X El Capitan):

  1. Open Utilities > Console.
  2. Click All Messages.
  3. Type Okta in the search field and press Enter to filter all Okta-related messages.


If running either of these operating systems:

  • macOS 10.12.6 (Sierra)
  • macOS 10.13.4 (High Sierra)
  1. Open Utilities > Console.
  2. Click your device under Devices in the left pane.
  3. Type Okta in the search field and press Enter to filter all Okta-related messages.


To view past logs for all supported operating systems:

  1. Open Utilities > Console.
  2. Go to /var/log > asl in the left pane.
  3. Select the date you want to search on.
  4. Type Okta in the search field and press Enter to filter all Okta-related messages.
