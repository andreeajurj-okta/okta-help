ServiceNow UD Provisioning Migration Guide

OVERVIEW

This document provides instructions to migrate your existing ServiceNow integration to use Universal DirectoryUniversal Directory enables you to store an unlimited amount of users and attributes from applications and sources like AD or HR systems. Any type of attributes are supported including linked-objects, sensitive attributes, and pre-defines lists. All of it accessible by all apps in our OIN catalog, over LDAP or via API. (UD)

WHAT'S NEW

The following features are new for ServiceNow UD

  • Support for ServiceNow Geneva version and above
  • User Schema DiscoveryAbility to import additional attributes to Okta support with unlimited custom attributes
  • Flexible user attribute mapping support
  • The Okta ServiceNow plugin is no longer required for provisioning

PROCEDURES

In order to migrate your existing ServiceNow integration to use UD, you need to configure a new instance, configure it to be as close as possible to your existing ServiceNow app instance, transfer your existing users to it, and finally copy over any existing mappings you may have. Here are the steps required:

Migration Steps

We have noted where the migration steps differ if your current ServiceNow app is configured with SWA or SAML SSO.

  1. Disable all provisioning features for your existing ServiceNow app.
  2. Configure a new ServiceNow UD app instance. See Configure a new ServiceNow UD app instance.
  3. Depending on how your current ServiceNow app integration is configured, we recommend two separate flow of migrations:

    • Migrate using Group Assignment

      Use this method if the following criteria are met, otherwise use Migrate using Import and Auto-confirm :

      • You have all users assigned via group level for your current ServiceNow - Eureka app instance.
      • All users in groupsGroups allow you to organize your end users and the apps they can access. Assigning apps to large sets of end users is made easier with groups. sorted by the following attributes and have the same values for them: Department, Cost Center and Location (if it is added - See Configure a new ServiceNow UD app instance)
      • All users in groups have the same values for attributes Time zone and Company on ServiceNow side. These attributes aren't supported by the old Service Now app, but are supported in new UD app

      Note: You should understand that the values of some attributes could be overwritten for all users during the migration, if this is not critical for you, go ahead to the migration flow using Group assignment.

    • Migrate using Import and Auto-confirm

Migrate using Group Assignment

Contact Okta Support to ensure your organization has the APPLICATION_ENTITLEMENT_POLICY feature flag enabled. This feature allows you to map user attributes instead of static values in group assignments (for example, user.department instead of Financial for all users in group).

Before migration, Okta recommends you try a test migration:

If your current ServiceNow app is not configured with Active Directory and all attributes are mapped to the Okta user profile

  1. Create a new test group in Okta.
  2. Create a new test user in Okta and set values for attributes mapped to ServiceNow (manager, department, cost center etc).
  3. Add this user to the test group.
  4. Assign the test group to your current ServiceNow app (make sure you have provisioning features enabled) and check the user is created in ServiceNow side with the correct attributes values.
  5. Disable provisioning for your current ServiceNow app.

  6. Assign the test group to your new ServiceNow UD app and map the same Active Directory attributes.

  7. Check the user is linked and the user's profile is updated in ServiceNow only for attributes that don't exist in the old Service Now app.

If your current ServiceNow app is configured with Active Directory:

  1. Create a new test group in Active Directory.
  2. Create a new test user in Active Directory and set up all attributes that are mapped from Active Directory to ServiceNow (manager, department, cost center etc).
  3. Add the new test user to the test group.
  4. Go to Okta and perform an import from Active Directory.
  5. Confirm the test user exists in Okta.
  6. Assign the test group with the test user to your old ServiceNow app.
  7. Make sure the user is created with correct attribute values.
  8. Disable provisioning for your old ServiceNow app.

  9. Assign the test group to your new ServiceNow UD app and map the same Active Directory attributes.

  10. Verify users and their profiles.

Notes:

If you ran into issues during your test migration, check the settings for inconsistencies, correct them and try again. If the test migration is successful, you can start the process of migration for all groups.

Be careful, we can not guarantee 100% success of the migration for all users without loss and rewriting the user's profile.

Now if testing is complete, you can continue with migration:

  1. Identify one or more group(s) in Okta which collectively include all users assigned to your current ServiceNow integration.
  2. Make sure you have disabled provisioning features for your current ServiceNow app.
  3. Go to your new ServiceNow UD app and assign the group.
  4. During assignment, set all group level attributes accordingly.

If you have blank values for some attributes for users in group, you may set Not Selected from the dropdown list, or you can specify the value you want.

Hide or Deactivate the old ServiceNow app

Now your new ServiceNow UD app is set up and assigned, we recommend you either Hide or Deactivate your old ServiceNow application tile from the end-user Okta console to avoid confusion. To do so:

Hide
  1. Open the old ServiceNow app, then select the General tab.
  2. Click Edit.
  3. Check Do not display application icon to users.
  4. Click Save.
Deactivate

In this case, all previously imported groups for old ServiceNow app will be deleted in Okta:

  1. Open the old ServiceNow app.
  2. Click the Active button, then select Deactivate.

  3. Click Save.

  1. In your existing ServiceNow UD app instance, select the Provisioning tab, then select To Okta in the left nav. In the User Creation & Matching section select Imported user is an exact match to Okta user if - Email matches:

End-User Migration flow

After you have assigned the new ServiceNow UD app (new application tile), all your end users need to do is click on the new application tile on their console. If they are able to login to ServiceNow they don't need to do anything else.

Migrating users with the same email address in ServiceNow

If you are migrating users with the same email address in ServiceNow, Okta recommends that you use single user assignment, as follows:

  1. Make sure you have disabled provisioning for your current/old Service Now app instance.
  2. Identify the users in Okta who have the same email address in both ServiceNow and Okta.
  3. Check all attribute values are the same for both Okta and ServiceNow users to avoid overwriting values.
  4. Assign these users to the new ServiceNow UD app instance.
  5. Check users in ServiceNow; the user profile should be updated only for the new attributes.
Migration for Pushed Groups
  1. Go to your old ServiceNow - Eureka and later releases app instance, unlink and delete the pushed group.
  2. Go to your new ServiceNow UD app instance and push this group.

Note:You will see the following error if you attempt to push the same group from you new ServiceNow app, without first deleting it from your old ServiceNow app:

How to verify your migration

You can check your migration as follows:

  • Check System Logs about importing users.
  • Verify that the number of users imported match the number of existing users for your existing ServiceNow app instance.
  • Login via the application tile for new ServiceNow UD app and check user's permissions.
  • Check your Dashboard for errors.
