Upgrade to Identity engine

Some features in Identity engine have a new UI and are configured differently than Okta Classic. Before you start your upgrade to Identity engine, make a note of the expected behavior in this section.

Administrator permissions

  • Super admin access is required to create sign-on policies and configure assurance and authenticators.

Identity engine UI changes

In Identity engine, the sign-in experience has changed for end users. New users register and activate their profiles in the sign-in widget instead of the welcome wizard, allowing you to fully customize and brand the onboarding experience with custom Okta-hosted sign-in pages.

Admin configuration for sign-on policies and rules has some new additions. During sign-in, end users authenticate through the primary and secondary authenticators among those you’ve made available in your policies.

Dependencies

  • If you use a custom Okta-hosted sign-in page, ensure that it uses version 4.3.0 (or later) of the sign-in widget.

  • Prior to the upgrade, existing orgs that upgrade to Identity engine should ensure that super admins enroll in either SMS or WebAuthn so that they can access the Admin app.

Limitations and known issues

  • The following authenticators are not supported:

    • Google Authenticator
    • RSA
    • Custom TOTP factor
    • Custom SAML factor
    • Security Question for MFA
    • Symantec VIP
    • Legacy YubiKey

  • The network condition All Networks is not supported in Okta sign-on policies.

  • Hardware authenticator possession factors are not yet available.

  • The password policy and MFA enrollment policy UI do not reflect the new definition of factor. SMS and Voice call show separately instead of represented as combined phone authenticator.

  • App sign-on policy rules don’t support user conditions. Existing rules that leverage an “exclude user” condition will not be migrated.

  • When completing their activation, new end users should set up their passwords before enrolling in a security question factor. Selecting a security question before setting a password results in an error and the admin is unable to resend the activation link.

  • When an app sign-on policy’s priority rule requires two-factor authentication every time, end users are not offered enrollable factors during self-registration. Instead, they receive this message: “Unable to fully authenticate into appInstanceId”

  • The 5-minute default lifetime for Email Magic Link and OTP is not currently configurable. Custom lifetimes that you set for the email factor do not migrate when you upgrade.

  • Users see a lock out message whenever they exceed their allowed number of sign-in attempts. This message appears even if admins have disabled Show lock out failures in the password policy.

  • The Remember Device by Default check box in the Okta Sign-on Policy rule (visible if you select per device for when to prompt for a secondary factor) doesn’t check the Remember me box by default for the user.

  • If one user already has an active session, and a different user tries to sign in through an Identity Provider-initiated flow, the second sign-in fails.

  • Security question can’t be used in conjunction with password to satisfy two-factor authentication in an app sign-on policy. Two-factor authentication in app sign-on policies requires two different factor types to be verified.

  • When the audience parameter is provided to the /authn API, the Okta Classic app sign-on policy is applied instead of the Identity engine app sign-on policy.

  • New sign-on notification emails for end users are currently not supported.

Safe mode

During maintenance or deployments, Okta servers may briefly go into safe mode. Safe mode can affect the end user’s MFA enrollment in the following ways:

  • If Okta servers are in safe mode when an end user attempts to access an app, the user must meet the app sign-on policy’s authenticator requirements.

  • If Okta servers go into safe mode while an end user is in the process of enrolling, an “Authenticator setup is not available” message appears. If the end user has already met the app sign-on policy’s authenticator requirements, they may skip enrollment and go directly to the app. If the app sign-on policy’s requirements have not been met, the user is advised to either return to the settings menu or sign out and try again later.

Behavior after upgrading to Identity engine

Note how the following features are affected once you upgrade to Identity engine:

Policies

  • Okta sign-on policies are preserved without any changes.
  • App sign-on policies are preserved and are migrated to the equivalent policy in the new policy UI.
    • If an app does not have an app sign-on policy, a default policy is created without any specific conditions.
  • Self-Service Registration policies are migrated to an equivalent User Profile Enrollment policy that applies to all applications.

MFA factors and enrollment

  • MFA factors that were enabled become enabled authenticators.
    • If an org previously uses email, SMS, or Voice for recovery but did not enable those factors as an MFA factor in their current org, these factors by default will not be available when signing in. Admins must enable those authenticators for SSO before end users may use them for signing in.
  • Factor enrollments for each user become authenticator enrollments:
    • SMS and Voice enrollments become Phone authenticators.
    • Password, Email and WebAuthn remain the same.
    • If a user is enrolled in both SMS and Voice with one phone number, they will have one Phone authenticator enrollment.
    • If a user has two phone numbers, they will have two Phone authenticator enrollments. Both Phone enrollments can use SMS and Voice methods.
    • If a user has one or more WebAuthn enrollments in their current org, all enrollments will be migrated.
    • If the user has one or more enrollments, they will be unable to enroll in additional WebAuthn enrollment.
    • Security questions used for authentication will be migrated to security question authenticator enrollments in Identity engine.
  • Voice, SMS, Email, and Security question methods for self-service password reset are migrated to authenticator enrollments:
    • Email addresses used for account recovery in the current org will be migrated as an email authenticator enrollment.
    • Phone numbers used for account recovery in the current org will be migrated as phone authenticator enrollments.
    • Security questions used for account recovery in current org will be migrated as a security question authenticator enrollment. Security questions cannot be used for signing in.

Password enrollment

  • Password enrollment is now a separate step during the sign-up process for end users.

Registration inline hooks

  • Registration inline hooks continue to work as expected when a new user attempts to register.

 

Turn off Identity engine

Should you choose to roll back to Okta Classic, note that some changes are not preserved when Identity engine is turned off. In these cases, the changes are reverted to the last known safe state, which minimizes disruption of the end-user experience.

  • Authenticators that are enabled for the org will not be converted back to their factor equivalents. Users who have enrolled these authenticators can no longer use those credentials.

  • App sign-on policies that you create with Identity engine are not preserved. If you turn off Identity engine, those rules are lost and your app sign-on policy rules revert to those that existed before enabling Identity engine.

  • Self-service registration reverts to the policy that existed before.

  • Okta sign-on policy’s Allow any option setting is not preserved. Password is required again.
